# Batch 6 デプロイ編 — ローカルのjarが世界から見えるURLになるまで

## ⚡ 結論チートシート

- 経路は **ローカル → GitHub → Render** の3点。pushすると自動でビルド・公開される
- 追加したのは4ファイル+1行: `Dockerfile`(ビルドと実行の設計図)、`.gitignore`、`.dockerignore`、`render.yaml`、`server.port=${PORT:8080}`
- Dockerfileは**マルチステージビルド**: Maven入りの重い環境でjarを作り、JREだけの軽い環境にjarだけ載せる
- 本番URLはHTTPSなのでWebSocketはwss:// — 前回の1行修正が今回の前提だった
- **重要な代償**: Render無料枠は15分アイドルでスリープし、復帰時に**メモリ上の対戦は全部消える**。Batch 0の「DBを使わない」判断の請求書がここで届く
- 更新の流れ: コード修正 → commit → push → 自動再デプロイ(CI/CDの入り口)

## 1. デプロイとは何か — 全体像

今までの「Eclipseで実行」は、マスターのPCの中でjarが動き、localhostだけがそれを知っている状態だった。デプロイとは、この実行を**インターネットから到達できる他人の計算機の上**に移すことである。今回の経路は3点を結ぶ。

```
[ローカル]                [GitHub]                    [Render]
 ソースコード  --push-->   リポジトリ(原本置き場)  --検知-->  ①Dockerfileの手順でビルド
                                                          ②できたイメージを起動
                                                          ③https://xxx.onrender.com で公開
```

ポイントは、**Renderに送るのはjarではなくソースコード**だということ。「どうビルドするか」はDockerfileに書いてあり、Renderはそれを読んで自分でビルドする。手元でビルドした成果物を手で運ぶのではなく、**原本(ソース)と手順(Dockerfile)を渡して、向こうで再現させる**——「消して困るものは原本管理が間違ってる」(target/の回で話した原則)の実践形であり、誰の環境でも同じものが作れることの保証でもある。

## 2. 追加した4ファイルの解説 ★

**Dockerfile** — 「このアプリを動かすOS環境ごと」の設計図。読みどころはマルチステージビルドで、ステージが2つある。前半(build)はMavenとJDKの入った重い環境でjarを作り、後半はJRE(実行環境)だけの軽い環境に**jarファイル1個だけ**をコピーする。Mavenもソースコードも本番イメージには存在しない。理由は2つ: イメージが小さいほど起動・転送が速い、そして**本番に不要な道具を置かないことは攻撃面の縮小**でもある。もう1つの仕掛けが「pom.xmlだけ先にCOPYして依存解決」の順序で、Dockerは各命令の結果をキャッシュするため、pom.xmlが変わらない限り依存ダウンロード(数分)が2回目以降スキップされる。**変わりにくいものを先に、変わりやすいものを後に**、がDockerfileの層設計の定石である。

**server.port=${PORT:8080}** — Renderは「このポートで待ち受けよ」を環境変数PORTで指示してくる。`${PORT:8080}`は「PORTがあればそれ、なければ8080」の記法で、**同じ設定ファイルのままローカルと本番の両方で動く**。環境ごとの違いはコードに書かず、環境変数で注入する——12要素アプリと呼ばれる本番運用の基本原則の、最初の一歩である。

**.gitignore / .dockerignore** — リポジトリに入れないもの(target/ = 生成物、.settings/等 = Eclipseの個人環境)と、Dockerビルドに送らないものの宣言。クローンした人はMavenインポートすれば環境ファイルが再生成されるので、原本だけを共有する。

**render.yaml** — Render上のサービス構成のコード化(Infrastructure as Code の入り口)。無くても画面から手動設定できるが、「構成もリポジトリに入っている」状態にしておくと再構築が1手になる。

## 3. 手順 ★

### 3-1. GitHubにリポジトリを作る

1. https://github.com でアカウント作成(無料)
2. 右上「+」→「New repository」→ 名前 `qte-battle`、**Private**を選択(認証なしアプリなのでソースは非公開推奨)→ Create

### 3-2. プロジェクトをpushする

GUI派なら **GitHub Desktop**(https://desktop.github.com)が楽:

1. インストールしてGitHubアカウントでサインイン
2. File → Add local repository → プロジェクトフォルダを選択(「create a repository」を促されたらそのまま作成)
3. 左下のSummaryに `Batch 6: デプロイ準備` と入れて「Commit to main」
4. 上部の「Publish repository」→ さっき作った`qte-battle`リポジトリへ(または新規作成でも可)

コマンドライン派なら(プロジェクトフォルダで):
```
git init
git add .
git commit -m "Batch 6: デプロイ準備"
git branch -M main
git remote add origin https://github.com/<ユーザー名>/qte-battle.git
git push -u origin main
```

いずれの場合も、`.gitignore`のおかげで target/ やEclipse設定はコミットに含まれない(GitHub Desktopの変更一覧に出てこないことを確認してみてほしい)。

### 3-3. Renderで公開する

1. https://render.com にGitHubアカウントでサインアップ(無料。カード登録不要)
2. ダッシュボード「New +」→「Web Service」→ GitHub連携を許可 → `qte-battle`リポジトリを選択
3. RenderがDockerfileを自動検出する。Instance Typeは **Free** を選択 → 「Deploy Web Service」
4. ビルドログが流れる(初回は5〜10分)。「Live」になったら `https://qte-battle-xxxx.onrender.com` が発行される
5. そのURLを開いてロビーが出たら成功。相手にURLを送り、部屋コードで合流すれば**世界のどこからでも対戦できる**

### 3-4. 以後の更新

コードを直したら commit → push するだけで、Renderが検知して自動で再ビルド・再公開する。これが**継続的デプロイ(CD)**の最小形であり、今後のバッチの差し替え作業は「zipを展開してpush」に変わる。

## 4. 本番で初めて見える設計の代償 ★

デプロイすると、今まで理屈だけだった設計判断の帰結が現実になる。

**スリープと状態の消失。** Render無料枠は15分アクセスがないとスリープし、次のアクセスで再起動する(復帰に1分弱かかる)。そして再起動すると——**進行中の対戦は全部消える**。部屋も、盤面も、メモリ上にしかないからだ。これはバグではなく、Batch 0で「試合状態は揮発してよい」と**意図的に**決めた設計の、目に見える代償である。設計判断には必ず請求書が来る。今回の請求額(たまに対戦が消える)は趣味アプリとして支払える範囲、というのが当時の判断で、それは今も妥当だと思う——が、もし許容できなくなったら「状態の永続化」という新しい章が始まる。

**スケールアウト不可。** このアプリはサーバ1台前提である。2台に増やすと、部屋Aはサーバ1のメモリに、部屋Bはサーバ2のメモリにあり、プレイヤー2人が別サーバに繋がれた瞬間に破綻する。singletonビーンのMapに状態を持つ設計は、**「1プロセス」という暗黙の前提**を持っていた。Render無料枠は1台なので今は問題にならないが、この制約を知っていることと知らないことの差は大きい。

**セキュリティの現在地。** 認証はなく、URLを知る人は誰でも部屋を作れる。playerIdのUUIDと部屋コードが実質の鍵で、友達と遊ぶ分には足りるが、不特定多数への公開を想定した作りではない。リポジトリをPrivateにしたのもこの割り切りの一部である。

## ✅ 理解確認(2問)

1. Renderのスリープ復帰後に対戦中の部屋が消える。これはBatch 0のどの設計判断の帰結か。また、消えないようにするには何が必要になるか。
2. Dockerfileで`COPY pom.xml .`と依存解決を`COPY src ./src`より**先**に書いている理由を、Dockerの層キャッシュの観点から説明せよ。
