# Batch 0 設計解説 — なぜゲーム状態をDBではなくメモリに持つのか

## ⚡ 結論チートシート

- 試合の状態は`GameState`(素のJavaオブジェクト)としてメモリ上に持つ。**DBは使わない**
- 置き場所はsingletonビーン`GameRoomManager`の`ConcurrentHashMap`。EM-Springで学んだ「singletonはステートレスに」の**意図的な例外**
- `GameState`一族はビーンに**しない**。試合ごとに生まれて消えるデータは、アプリと同寿命のビーンと寿命が合わないため
- パッケージは「不変マスタ(`master`)/可変な試合状態(`game`)/部屋の台帳(`room`)/入口(`web`)」の4分割
- 動作確認: Eclipseにインポート → `QteBattleApplication`を実行 → http://localhost:8080 で46枚のカード一覧が表示されれば成功

---

## 1. DBを使わない、という判断 ★

EM-Springでは「データはDBにあり、Entityを介して読み書きする」が大前提だった。部署を登録すれば、サーバを再起動しても部署は残る。**残ることに価値がある**データだからである。

対戦カードゲームの試合状態は性質が逆になる。

| | EM-Springの部署データ | QTEの試合状態 |
|---|---|---|
| 寿命 | 永続(何年も残る) | 1試合(数十分で消えてよい) |
| 再起動後 | 残っていないと困る | 消えても許容(趣味アプリの割り切り) |
| 更新頻度 | 低い(1操作=1更新) | 極めて高い(1アクションで盤面のあちこちが変わる) |
| 形 | 表形式(行と列) | 深い木構造(試合→プレイヤー→ゾーン→ミニオン→修正値) |

試合状態をDBに置くと、1回の「攻撃」処理のたびに複数テーブルへのUPDATEとその読み直しが走る。得られる対価は「再起動しても試合が残る」ことだが、今回それは要らない。**要らない対価のためにコストを払わない**、が判断の中身である。

なお、消えては困るデータ(カードマスタ)は別扱いだが、これもDBではなくJSONにした。理由は3章で述べる。

## 2. 状態はどこに「置かれて」いるのか ★

DBがないなら、試合状態はメモリのどこに存在し続けるのか。答えは`GameRoomManager`にある。

```java
@Component
public class GameRoomManager {
    private final Map<String, GameRoom> rooms = new ConcurrentHashMap<>();
}
```

`GameRoomManager`はsingletonビーンであり、アプリケーションと同じ寿命を持つ。その**フィールドにMapをぶら下げる**ことで、リクエストが終わっても試合状態が生き続ける。HTTPリクエストの寿命(処理が終われば消える)とビーンの寿命(アプリと同寿命)のギャップを利用した置き場所である。

ここでEM-Springの学習内容と正面衝突する。**「singletonビーンにフィールドで状態を持たせるな(ステートレスにせよ)」**という原則だ。あの原則の根拠は「全リクエストが同じインスタンスを共有するため、書き換え合うと壊れる」だった。今回は根拠の側に対処して、例外を成立させている。

1. **共有が目的そのもの**である。部屋IDを知る2人が同じ`GameState`に到達できることが機能要件
2. **同時書き込みは構造で防ぐ**。台帳への出し入れは`ConcurrentHashMap`が守り、個々の試合の変更は`GameRoom`の`lock`による`synchronized`で「1部屋につき同時1操作」に直列化する(実際に使うのはBatch 1)。部屋ごとに別ロックなので、隣の部屋の試合を待たせない

つまり原則は「禁止」ではなく「共有と競合を制御できないなら持つな」であり、制御を設計に組み込んだ上での保持は正当、という整理になる。

もうひとつ対になる判断がある。`GameState`や`PlayerState`を`@Component`にしなかったことだ。ビーンにする動機が何もない——DIで注入したい相手はいないし、singletonでは全員が同じ1試合を共有してしまう。「試合ごとに`new`されて、試合が終われば参照が切れてGCされる」素のオブジェクトが寿命として正しい。**Springのビーンは「アプリの部品」のためのもので、「アプリが扱うデータ」はビーンにしない**。EntityをビーンにしなかったEM-Springと同じ線引きである。

## 3. 不変マスタと可変インスタンスの分離 ★

パッケージ`master`と`game`の分離が、このプロジェクトの背骨になる。

- `master.CardMaster`: カード1種類の定義。ゲーム中**絶対に変化しない**。recordで実装し、変更手段を型レベルで持たない
- `game.MinionInstance`: 場に出た1体。`CardMaster`への参照と、**変化する状態だけ**(累積ダメージ・付与キーワード・修正値・攻撃回数)を持つ

同じ「ディープシー・シャーク」が2体並んだとき、マスタは1個を共有し、個体状態は2個それぞれが持つ。個体は`instanceId`(UUID)で識別する——「どちらのシャークに攻撃するか」をカード名では指定できないからである。

カードマスタの置き場所をDBでなくJSONにしたのは、このデータが**更新系を一切持たない**ためだ。DBの価値(トランザクション・同時更新・柔軟な検索)がどれも要らず、必要なのは起動時に1回読むことだけ。`CardMasterRepository`は名前こそRepositoryだが、実体はコンストラクタでJSONを読み込んだ`Map`の表引きである。コンストラクタで読む(=読み込み失敗ならアプリが起動しない)のは、EM-Springの`ddl-auto: validate`と同じ発想——**データの問題は実行時ではなく起動時に落とす**。

## 4. Batch 0に意図的に「まだ無い」もの ★

骨組みの段階で作り込まない判断をしたものを明示しておく。

- **エフェクトシステム**: `CardMaster`は効果テキストを文字列のまま持つ。構造化はBatch 2。ただし受け皿になる枠(`StatModifier`の期限×演算、`grantedKeywords`、ダメージと破壊判定の分離を前提にした`damage`累積方式)は先に切ってある
- **動的ステータスの評価**: `getEffectiveAttack()`は固定値のSET→ADD合成のみ。「手札枚数を参照」のような動的値はエフェクトシステム側の責務にする
- **WebSocket・画面**: Batch 1。今の`HomeController`はマスタ読み込みの動作確認専用

## ✅ 動作確認手順

1. Eclipseで File → Import → Existing Maven Projects → 展開した`qte-battle`フォルダを選択
2. 初回はMavenの依存ダウンロードが走る(数分)
3. `QteBattleApplication`を右クリック → Run As → Java Application(またはSpring Boot App)
4. ブラウザで http://localhost:8080 を開く → **文明別に46枚のカード一覧**が表示されれば、マスタ読み込み・Thymeleaf・DI がすべて機能している
5. おまけ: `CardMasterLoadTest`を Run As → JUnit Test で4件グリーンになることも確認できる

## ✅ 理解確認(2問)

1. `GameRoomManager`はsingletonビーンなのにフィールドに可変のMapを持っている。EM-Springで学んだ原則に反しないと言える理由を、「寿命」と「競合」の2語を使って説明せよ。
2. `GameState`を`@Component`にした場合、2組のプレイヤーが同時に対戦を始めると何が起きるか。
