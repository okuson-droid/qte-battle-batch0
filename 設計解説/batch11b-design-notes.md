## 1. 28枚中19枚は、11aが既に終わらせていた ★★

引き継ぎ書を読んだ時点でまず驚いたのがこれである。光文明のカードを1枚ずつ
「`CardEffectRegistry` に何を書くか」で仕分けていったところ、次の19枚は
**何も書く必要がなかった。**

```
平和の結界・煌めきの盾・創世神・ゾディアック・大天使ミカエル      → RuleGuardsが既に見ている
正義の御盾・断罪の大天使・秩序の執行官・戒律の聖堂騎士            → 同上
唱導の聖騎士・英知の水晶・戒律のガーディアン・詠唱の宝珠(コスト分)→ StatCalculatorが既に見ている
ホーリー・ガーディアン・聖域の司祭・ライト・シールド              → 【守護】【知識】だけの素の効果
```

これはバグでも手抜きでもない。11aの狙いが「判定点を先に作る」ことだったので、
**カードの登録(11b)は、判定点にIDを1つ通すだけで済むように設計されていた。**
実際に確認したのは、たとえば大天使ミカエルなら

```java
// RuleGuards.java (11aで既に書かれていたコード)
private static final String MICHAEL = "QTE-0004";
...
if (MICHAEL.equals(target.getMaster().id())) {
    return cause == DestructionCause.COMBAT; // 戦闘破壊だけ無効化
}
```

という分岐が**カード名で直書きされている**こと、そしてそのIDがqte-cards.jsonの
`QTE-0004` と一致していることだけだった。ここで実際にコードを書いていたら、
11aの判定層の意味がなかったことになる。**「今回は何もしない」と判断することも、
設計を検証する作業の一部である。**

残り9枚(聖域の案内人・光の召喚士・降臨の伝道師・運命のリセット・
ホーリー・シグナル・聖光の武装解除・神の福音・聖なる降誕の儀式・光の戒め)
と、リーダー2枚(断罪の聖導者・聖光の守護聖)だけが、このバッチで実際に手を動かした部分である。

## 2. ウェポンという「1人1枚」の対象 ★★

聖光の武装解除は「ウェポンを1枚破壊する。自分のウェポンも選べる」。
これまでの対象選択(`Kind`)は HAND・MINION・MANA・TRASH の4種類だったが、
どれも「複数の候補からインスタンスIDで選ぶ」形をしている。ウェポンは違う。
**各プレイヤーに最大1枚しか存在しない**ため、選ぶべきはインスタンスではなく
「自分の分か、相手の分か」という**側**だけである。

```java
public record TargetChoice(..., List<String> weaponSides) // "SELF" / "OPPONENT"
```

インスタンスIDの代わりに文字列2値で表現したのは、既存の4種類が
「候補のリストから選ぶ」ために設計されていて、候補が高々2つ(しかも
存在するかどうかも不定)のウェポンにそのまま当てはめると、
存在しないインスタンスIDの空リストを作るような無理が生じるからである。

検証側(`GameService.validateTargets`)も、既存の `findOnSide` は流用せず
その場で `"SELF"`/`"OPPONENT"` を `PlayerState` に変換する数行を足すだけで済んだ。
**新しい種類の選択を1つ増やすコストが、既存の4種類を無理に拡張するより低かった。**

## 3. 潜伏の初めての例外 ★★

ホーリー・シグナルは「相手の場で最も攻撃力の高いミニオンを破壊する。
【潜伏】持ちであっても破壊できる」。総則には「【潜伏】は相手のカードや
能力の対象にならない」とあるので、これはテキストによる**上書き**である。

これまで潜伏の判定は `validateTargets` のMINION分岐に無条件で書かれていた。

```java
if (tm.owner() != player && tm.minion().hasKeyword(Keyword.STEALTH)) {
    throw ...
}
```

この一行を消してホーリー・シグナル専用に迂回させると、次に「普通に潜伏を
尊重する」カードを書くときにまた同じ分岐を足す羽目になる。そこで
**フィルタの並び(`Filter[]`)に「潜伏を無視してよい」という指示そのものを
乗せる**ことにした。

```java
boolean stealthBlocked = tm.owner() != player && tm.minion().hasKeyword(Keyword.STEALTH)
        && !req.filters().contains(TargetSpec.Filter.IGNORES_STEALTH);
```

`Filter` は本来「選べる候補を絞り込む条件」であり、`IGNORES_STEALTH` は
絞り込みではなく指示なので、意味的にはやや異物である。それでも新しい
`Requirement` のフィールドを1つ増やす(全カードのコンストラクタ呼び出しに影響する)より、
既存の「フィルタの列」という差し込み口を再利用するほうが影響範囲が小さいと判断した。

## 4. 「盤面から自動で決まる、決まらなければ選ぶ」除去 ★

ホーリー・シグナルの対象は「相手の場で最も攻撃力の高いミニオン」であり、
プレイヤーが自由に選ぶわけではない。**タイでなければ選択の余地がなく、
タイのときだけ実質的な選択になる。**

これを対象選択の枠組みの中で表現するには、盤面全体を見ないと判定できない
条件が要る。これまでの `Filter` はカード1枚(と現在HP)だけを見て判定できたが、
今回は「相手の場の最大値」という**他のミニオンとの比較**が要る。

```java
case HIGHEST_ATTACK_OPPONENT -> {
    PlayerState targetSide = state.opponentOf(player.getPlayerId());
    int max = targetSide.getMinionZone().stream()
            .mapToInt(m -> stats.effectiveAttack(state, targetSide, m))
            .max().orElse(Integer.MIN_VALUE);
    ...
}
```

そのため `checkFilter` に `state` と `player` を新たに渡すよう変えた
(呼び出し元は3箇所とも既に両方を持っていたので、引数を足すだけで済んだ)。
クライアント側も同じ判定をJSで再現している(`attack` は実効値がビューに
既に乗っているため、サーバ側の `StatCalculator` を呼び直す必要はない)。

## 5. 山札を覗く、初めての「非公開情報の一時公開」 ★★★

降臨の伝道師は「山札の上から4枚を表向きにする。【守護】を1体場に出し、
残りは山札の下へ」。ここでの難しさは、**候補が「登場した瞬間」にしか
分からない**ことである。手札・場・マナ・墓地はどれもプレイ宣言の時点で
中身が見えているので、クライアントは既存の選択UIをそのまま出せた。
山札はそうはいかない。

0体/1体のときは選択の余地がないので、その場で自動的に解決してしまう。

```java
if (guardIndexes.isEmpty()) {
    ctx.actions().returnToBottomOfDeck(ctx.owner(), revealed);
    return; // 不発
}
if (guardIndexes.size() == 1) {
    resolveMissionaryChoice(ctx, revealed, guardIndexes.get(0)); // 自動決定
    return;
}
```

2体以上のときだけ、`PlayerState` に新しく持たせた `pendingReveal`
(公開した4枚そのもの)に選択待ちの状態を持たせ、専用のWSエンドポイント
`resolve-reveal` の到着を待つ。

```java
private final List<String> pendingReveal = new ArrayList<>();
```

**この間、他の操作は一切受け付けない。** 選び終える前にターンを進められると
辻褄が合わなくなるため、`requireTurnPlayer` (ほぼ全ての操作が経由する共通判定)
に「`pendingReveal` が空でなければ弾く」という1行を足した。10箇所ある
呼び出し元を個別に触らずに済んだ一方、選択そのものを解決する
`resolveRevealChoice` は意図的にこの共通判定を経由させていない。
経由させると「選び終えるまで塞ぐ」判定に、選ぶための操作自身が
引っかかってしまうからである。

クライアント側も、手札・場・マナ・墓地のどの見た目とも違う専用のUI
(`reveal-area`)を新設した。既存の対象選択(`pending`)とは別の変数として
`view.you.pendingReveal` を直接見て描画するだけの、小さな独立したUIにしている。

## 6. 見つけた2つの既存の穴 ★

カードを1枚ずつ実装していく過程で、光文明とは無関係の**既存のバグ**を2つ見つけた。

**① 断罪の聖導者のスペル封じが、禁忌カードに効いていなかった**

`playCard`(通常のスペル使用)は `guards.spellDenial` を見ていたが、
`playTabooCard`(禁忌デッキからの使用)には同じ判定がなかった。
断罪の聖導者のnotesには「禁忌デッキ由来のスペルも封じられる」と
明記されていたため、実装しようとして初めて気づいた。`playCard` と
全く同じ3行を `playTabooCard` にも足して揃えた。

**② デッキビルダーの実装済み文明リストに、闇(Batch 10b)が入っていなかった**

```js
// 修正前(deck-builder.js)
const IMPLEMENTED_CIVS = ['WATER', 'FIRE'];
```

光を追加しようとして一覧を開いたら、10bで実装したはずの闇が
最初から漏れていた。両方まとめて直した。

**この2つに共通するのは、「新しい機能を足すために既存のコードを読む」
という作業そのものが、レビューの機会にもなっている**ということである。
機械的チェックだけでは見つからない種類の抜けだった。

## ✅ 理解確認

**問1.** 光文明28枚のうち19枚は `CardEffectRegistry` に何も書かなかった。
これは手抜きか、設計判断か。理由も答えよ。

<details><summary>答え</summary>

設計判断である。11aで判定層(`RuleGuards`・`StatCalculator`)を先に用意し、
そこにカードIDを直書きする形にしておいたため、11bで実際にコードを書く
必要があるのは「判定層では表現できない、能動的な処理を伴うカード」だけになる。
何も書かずに済んだことは、11aの設計が意図通り機能した証拠として確認すべき点であり、
やり残しではない。
</details>

**問2.** `IGNORES_STEALTH` を `Filter` の列に乗せたのはなぜか。
`Requirement` レコードに `boolean ignoresStealth` を足す方法と比べて何が違うか。

<details><summary>答え</summary>

`Requirement` は record であり、フィールドを1つ増やすとその不変コンストラクタと
3つの静的ファクトリ(`of`/`filtered`/`upTo`)の呼び出し全て(=登録済みの
全カードの対象指定)に影響する。既存の「フィルタの列」という差し込み口を
再利用すれば、影響範囲はホーリー・シグナル1枚と、判定を読む側の
数行だけで済む。
</details>

**問3.** 降臨の伝道師で2体以上の【守護】が公開されたとき、
`resolveRevealChoice` は `requireTurnPlayer` を経由していない。理由は何か。

<details><summary>答え</summary>

`requireTurnPlayer` には「`pendingReveal` が空でなければ弾く」という判定が
入っている。もし `resolveRevealChoice` がこれを経由すると、選択待ちの状態そのものが
その選択を完了させる操作自身をブロックしてしまい、詰んでしまう。
選択待ちを解消する操作だけは、この共通判定の外に置く必要がある。
</details>
