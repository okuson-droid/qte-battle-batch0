# Batch 5 設計解説 — マリガンとUI改善: 「見せてよい情報」と「進行の主導権」

## ⚡ 結論チートシート

- マリガンを総合ルール5章の**手順順序通り**に実装した(初期ドロー→マリガン→ピュア・エレメント付与→開始)。順序に従った結果、「デッキ外カードがマリガンで山札に混入する」バグが構造的に不可能になった
- 墓地・マナ・リーダー能力の閲覧UIを追加。**何が見えるかはサーバのビューが既に決めている**ため、クライアントは届いているものを表示するだけ(相手の裏向きマナは最初からデータが来ない)
- 自動進行はクライアント側の判断で実装した。「とれるアクションがない」の判定は近似であり、判定を間違えても**サーバの検証が最後の壁**なので安全に軽く作れる
- ウェポンが光らないのは指摘通りバグ(プレイ可否条件の足し忘れ)。修正済み
- 動作確認: マリガン→自動進行トグル→墓地/マナ/リーダー閲覧→ウェポンの光り

---

## 1. マリガン — ルールの手順順序が設計を守る ★

マリガンの実装で一番重要な判断は、実は選択UIではなく**処理の順序**だった。総合ルール5章はセットアップを「②初期ドロー → ③マリガン → ④後攻にピュア・エレメント → ⑤開始」の順に定めている。当初のBatch 1実装では(マリガンがなかったので)②と④を同時にやっていたが、マリガンを足すにあたって**ルールの順序に忠実に直した**。

これが単なる律儀さでない理由: ピュア・エレメントはデッキ外から配られるカードであり、もしマリガン前に手札にあると「引き直しでデッキに混ぜてしまう」事故が起きうる。選択画面で除外する防御コードを書く手もあるが、**配布をマリガンの後に移せば、防御コードそのものが不要になる**。仕様の手順に沿うことがバグの発生源を消す——「エラーを検出する」より「エラーを表現不能にする」ほうが強い、という設計原則の小さな実例である。

もうひとつ、マリガンの「戻して引き直す」処理はBatch 3の手札除去とまったく同じ落とし穴(インデックスの揮発)を持つため、同じ対策(ID確定→降順除去)を使っている。一度学んだパターンが別の機能でそのまま再利用される様子も見どころだ。

セットアップの状態遷移は「WAITING → SETUP(先後選択待ち) → SETUP(マリガン待ち) → PLAYING」となった。SETUP内の2段階は新しいステータス値ではなく「`firstPlayerId`が入っているか」で区別している。状態が増えるたびenumを増やすか、既存データから導出するか——ここでは遷移が単純なので導出を選んだ(段階がさらに増えるならenum化を検討する)。

## 2. 閲覧UI — 「見せてよいか」は既に決まっていた ★

墓地・マナ・リーダー能力の閲覧(指摘1・3・4)の実装は、実はクライアント側の表示だけでほぼ完結した。なぜなら**情報の可視性はBatch 1のビュー設計で既に決まっている**からだ。

- 墓地は公開情報 → `trashCardNames`を両者のビューに追加(サーバの1行)
- マナの状態(タップ/表裏)は公開、**相手の裏向きマナの中身だけ非公開** → 既存の`ManaView`がまさにその形(相手の裏向きは`name=null`で届く)。クライアントは`name || '(裏向きのカード)'`と表示するだけで、指摘4の要件が満たされる
- リーダーの効果テキストは公開(リーダーは表向きで公開されるカード) → `leaderText`を追加

もし「表示するかどうか」をクライアントのif文で制御していたら、この改修は「見せてはいけない情報を見せていないか」の再点検を伴っただろう。**送る段階でフィルタする**方針(Batch 1解説3章)の配当がここで出ている。データが来ていないものは、どうやっても表示できない。

## 3. 自動進行 — 近似判定をクライアントに置ける理由 ★

自動進行(指摘2)は「そのフェイズでとれるアクションがなくなったら自動で次へ」。この「とれるアクションがあるか」の判定をどこに置くかが設計の分かれ目で、今回は**クライアント側の近似判定**にした。

理由は2つ。第一に、この判定は**間違えても壊れない**。進めるべきでないのに進めた場合はプレイヤーが手動モードに戻せばよく、進めるべきなのに進めなかった場合は「次のフェイズへ」を押せばよい。ゲームの正しさはサーバの検証が守っているので、利便性レイヤーの判定ミスは不便どまりで済む。第二に、判定材料(手札のコストとMP、攻撃可否フラグ)はすべてビューに既に載っている。サーバに新しいAPIを生やす必要がない。

正確には近似で、既知の限界がひとつある: バトルフェイズで「攻撃可能なミニオンはいるが、合法な対象が存在しない」(相手が威圧だけ等の稀な盤面)ときに自動で進まない。厳密化するなら対象の合法性まで舐める必要があるが、稀なケースのために判定を複雑にするより「その時は手動で進める」でよい、と割り切った(コードのコメントに明記)。

実装の細部: 自動送信は同じ(ターン, フェイズ)につき1回に抑止している(`lastAutoKey`)。もしこの抑止がないと「送信→サーバが何かの理由で拒否→ビュー再配信→また送信」のループが起きうる。**自動でメッセージを送る機能には必ず再送抑止を付ける**——WebSocketに限らない、イベント駆動の定石である。

## 4. ウェポンが光らない件 — バグの分類

指摘5は正真正銘のバグで、原因は`createHandCardEl`のプレイ可否条件が`MINION || SPELL`のままだったこと(Batch 4でウェポンを解禁したときの足し忘れ)。修正は条件を「LEADER以外」に反転した——**許可リストに追加し続けるより、禁止リストで書くほうが追加漏れに強い**場合の例である(カードタイプは有限で、プレイ不能なのはLEADERだけと言い切れるため)。

このバグの教訓はもうひとつある。サーバは正しくウェポンのプレイを受け付けていたので、これは「表示だけの」バグだった。UIの補助情報とサーバの判定を分離した設計(Batch 1)のおかげで、バグの影響範囲が「光らない」に閉じ込められていた——同じ足し忘れがサーバ側にあったら「プレイできない」バグになっていた。

## ✅ 動作確認手順

1. 差し替えて起動、2タブで対戦開始
2. **マリガン**: 先後選択の後、両タブにマリガンUIが出る。引き直したいカードをクリック(紫枠)→「この内容で確定」。「引き直さない」で即確定。両者完了後に後攻へピュア・エレメントが配られ、第1ターン開始
3. **墓地閲覧**: 「墓地: N枚」クリック → 一覧モーダル。相手側も見える
4. **マナ閲覧**: 「マナN枚」クリック → 状態(タップ/表裏)付き一覧。**相手の裏向きだけ「(裏向きのカード)」**になることを、怨霊…はまだ無いので流転の書の還元マナ等で確認
5. **リーダー能力閲覧**: 「リーダー能力を確認」→ 効果テキストのモーダル。相手のリーダーも見える
6. **自動進行**: 「進行: 手動」をクリックして「自動」へ。手札が空/MP不足/攻撃済みのフェイズが約0.7秒で自動的に進む。アクションが残っているフェイズでは止まる
7. **ウェポン**: メインフェイズでMPが足りるウェポンの枠が緑に光る

## ✅ 理解確認(2問)

1. 「相手の裏向きマナを確認できない」ことを保証しているコードは、クライアントとサーバのどちらにあるか。その場所を1つ挙げよ。
2. 自動進行の判定をサーバ側に置いた場合(サーバがアクション有無を判定して勝手にフェイズを進める)、クライアント実装と比べてどんな問題が新たに生じるか。1つ挙げよ(ヒント: プレイヤーの「あえて何もしない」)。
