## 1. 闇と光では、必要になるものが逆だった ★★

Batch 10a では `GameActions` にメソッドを9個足した。闇のカードが要求したのは
「山札から墓地へ送る」「裏向きマナを表に戻す」といった**新しい操作**だったからである。

光文明を28枚ぜんぶ読んで、同じつもりで「何の操作が足りないか」を探したが、
ほとんど出てこなかった。代わりに出てきたのはこういうテキストである。

```
平和の結界     Attackが3以上のミニオンは攻撃できない
煌めきの盾     このカードは攻撃できない
創世神         リーダーを攻撃できない
ゾディアック   相手のリーダーは攻撃できない
大天使ミカエル 戦闘では破壊されない
聖光の守護聖   相手の効果では破壊されない
断罪の大天使   引く代わりに墓地に置く
断罪の聖導者   スペルを唱えられない
秩序の執行官   特殊召喚を行えない
戒律の聖堂騎士 サブフェイズを行えない
```

**すべて「既にある処理を止める・置き換える」ものである。** 必要なのは新しい操作ではなく、
既存の処理の中に**判定する点**を作ることだった。

そこで `RuleGuards` を新設し、判定を6箇所に集約した。

| 判定点 | 呼ぶ場所 | 差し込まれるカード |
|---|---|---|
| 攻撃宣言 | `GameService.validateAttack` / `leaderAttack` | 平和の結界・煌めきの盾・創世神・ゾディアック |
| 破壊 | `GameActions.destroyMinion` / `checkDestruction` | 大天使ミカエル・聖光の守護聖 |
| リーダーへのダメージ | `GameActions.damageLeader` | 正義の御盾 |
| ドロー | `GameActions.drawCards` | 断罪の大天使 |
| カードの使用 | `GameService.playCard` / `specialSummon` | 断罪の聖導者・秩序の執行官 |
| フェイズ進行 | `GameService.nextPhase` | 戒律の聖堂騎士 |

### 判定層は例外を投げない

`RuleGuards` の攻撃判定は、**理由の文字列を返す**(攻撃できるならnull)。

```java
String denial = guards.minionAttackDenial(state, player, attacker, targetIsLeader);
if (denial != null) {
    throw new IllegalStateException(denial);
}
```

例外を投げるのはルール本体(`GameService`)の役目であり、判定層は判定に徹する。
この分け方には具体的な利益がある——**同じ判定をビュー生成からも呼べる**。

```java
// GameViewBuilder
boolean canAttackMinion = attackerSide
        && guards.minionAttackDenial(state, owner, minion, false) == null;
```

これまでビュー側は、召喚酔いと速攻・突進の条件を**独自に書き写して**いた。
つまり同じルールが2箇所に書かれており、平和の結界を実装した瞬間に
「盤面では攻撃できるように見えるのに、押すと弾かれる」というズレが生まれるところだった。
判定を1つにしたことで、このズレが構造的に起こらなくなる。

## 2. 「破壊されない」を誰の効果か区別する問題 ★★

聖光の守護聖のテキストは「**相手のカードや能力の効果で**破壊されない」である。
自分の効果(生贄など)による破壊は防がない。ところが破壊処理は、
「誰の効果が原因か」という情報を持っていない。

```java
public void destroyMinion(GameRoom room, PlayerState owner, MinionInstance minion,
        DestructionCause cause)
```

`owner` は**破壊されるミニオンの持ち主**であって、破壊した側ではない。
正攻法は全ての破壊呼び出しに実行者を足すことだが、闇文明だけで
`destroyMinion` の呼び出しは20箇所以上あり、カード側の記述が全部変わる。

そこで推定を使った。

> 効果による破壊で、破壊されるミニオンの持ち主が**ターンプレイヤーでない**なら、
> その破壊はターンプレイヤー(=持ち主から見た相手)の効果によるものである

```java
boolean causedByOpponent = !owner.getPlayerId().equals(state.getTurnPlayerId());
```

**現在のカードプールでは、この推定は必ず正しい。** 自分のミニオンを破壊する効果
(絶望の連鎖・生贄を求める邪鬼・死者蘇生・這い寄る生霊の自壊)は、
すべて自分のターンに発動するからである。

これは妥協である。だから妥協であることと、破綻する条件を
`RuleGuards` のJavadocに明記した。「相手のターン中に自分のミニオンを自ら破壊するカード」が
登場した瞬間に、実行者を引数で渡す方式へ切り替える必要がある。

**推定を使うこと自体は悪ではない。悪いのは、推定したことを忘れることである。**

### ミカエルは HP0 のまま場に残る

「戦闘では破壊されない(ダメージは受ける)」を素直に実装すると、
HPが0以下のミニオンが盤面に残る。これはバグではなく仕様である
(発注者確認済み)。次に効果ダメージを受けるか、効果破壊されれば普通に処理される。

破壊原因の区別は Batch 10a で `DestructionCause` として導入済みだったので、
判定は1行で済んだ。**闇のボーン・コレクター(戦闘で破壊されたら1枚引く)のために
作った区別が、光の防御にそのまま流用できた。**

## 3. 期限に種類ができた ★

これまで「一時的な効果」はすべて `thisTurnAuras` に入れ、ターン終了時に
`clear()` していた。期限が1種類しかなかったので、それで足りていた。

光文明には2つの新しい期限が現れる。

```
詠唱の宝珠   : 次にスペルを唱えるまで(ターンをまたぐ・使ったら消える)
聖光の守護聖 : 次の相手のターン終了時まで(自分のターンをまたぐ)
```

素直にやるなら「宝珠用のboolean」「守護聖用のint」を `PlayerState` に足せばよい。
短く書けるし、動く。それでもやめたのは、**期限を管理するコードが散らばる**からである。
フィールドを足すたびに、`endTurn` の中に「これを消す」という行が1本ずつ増えていく。

そこで期限を型にした。

```java
public record PersistentAura(String cardId, Expiry expiry, int expiresAfterTurn) {
    public enum Expiry { AFTER_TURN_NUMBER, ON_NEXT_SPELL }
}
```

消す処理は2箇所だけで済む。

```java
// ターン終了時: ターン番号で切れるものを落とす
p.getPersistentAuras().removeIf(aura ->
        aura.expiry() == Expiry.AFTER_TURN_NUMBER && state.getTurnNumber() >= aura.expiresAfterTurn());

// スペル使用時: 「次にスペルを唱えるまで」のものを使い切る
player.getPersistentAuras().removeIf(aura -> aura.expiry() == Expiry.ON_NEXT_SPELL);
```

後者の**置き場所には注意が要る**。コストの計算(`effectiveCost`)が終わった後でなければ、
軽減が乗る前に効果が消えてしまう。コード上は数行の差だが、
プレイヤーから見れば「宝珠を割ったのに安くならない」という不具合になる。

## 4. ドローの置換 — 「引く」の意味を分解する

断罪の大天使は「相手はカードを**引く代わりに**そのカードを墓地に置く」。
これを実装するには、ドローという操作を分解する必要がある。

```
ドロー = ① 山札の一番上を取り除く + ② それを手札に加える
```

置換されるのは②だけで、①は同じように起きる。だから山札は減るし、
山札が尽きれば敗北判定も普通に行われる。

```java
String cardId = player.getDeck().pollFirst();
if (cardId == null) { /* 山札切れで敗北 */ }
boolean replaced = guards.isDrawReplaced(state, player, player.getDrawnCountThisTurn());
player.setDrawnCountThisTurn(player.getDrawnCountThisTurn() + 1);
if (replaced) {
    player.getTrash().add(cardId);   // ← ②だけが変わる
    continue;
}
player.getHand().add(cardId);
```

枚数の数え方は「そのターンに引いた通算」で、ターン開始時の通常ドローも
1枚目に含む(発注者確認済み)。カウンタは `startTurnReset` で0に戻る。

なお、**闇の「墓地に置く」(ミル)と行き先は同じでも意味が違う**。
ミルは自分から山札を削る能動的な操作、こちらは相手のドローを潰す置換である。
同じ `trash.add` でも、通る経路が違えば別の概念として扱う。

## 5. リーダーへのダメージ軽減

`damageLeader` の先頭に軽減を挟んだ。ここで1つ判断がある。

```java
int reduced = guards.reduceLeaderDamage(room.getGameState(), player, amount);
if (reduced <= 0) {
    room.addLog("ダメージは軽減されました");
    return;   // ← 被ダメージトリガーも発火しない
}
```

**軽減後が0なら「ダメージを受けなかった」ものとして扱う。**
火文明には「リーダーがダメージを受けたら」誘発するカード(火炎の狂信者・反転の炎鏡)があり、
0ダメージで誘発させるかは解釈の分かれるところである。
1ダメージが正義の御盾で0になったとき、それを「受けた」と数えるのは不自然と判断した。

## ✅ 動作確認手順

11a は内部構造の変更であり、光文明のカードはまだデッキに入らない。
したがって確認すべきは「壊していないこと」である。

1. `mvn spring-boot:run` で起動、2タブで対戦開始
2. **召喚酔いのミニオンが攻撃できない**こと、**【速攻】は殴れる**こと、
   **【突進】はミニオンだけ殴れる**こと → 攻撃判定を移設した箇所の回帰
3. 盤面のミニオンが**攻撃可能なときだけハイライトされる**こと
   → ビューの判定をサーバ側と共通化したため
4. **氷結の杖**で凍結させたミニオンが次のターン攻撃できないこと
5. **【守護】【威圧】【貫通】**の対象制限が今までどおり働くこと
6. ドローが正常に行われ、山札切れで敗北すること(ドロー処理に分岐を足したため)
7. リーダーへのダメージが通り、火文明の被ダメージトリガーが誘発すること
8. 闇のカード(破壊まわり)が今までどおり動くこと

## ✅ 理解確認

**問1.** ビューの攻撃可否判定を `RuleGuards` に統一した。
統一しなかった場合、平和の結界を実装したときに何が起きるか。

<details><summary>答え</summary>

ビュー側は独自に「召喚酔い・速攻・突進」だけを見ているため、Attack3以上のミニオンも
攻撃可能としてハイライトされる。プレイヤーが押すとサーバ側で弾かれ、
「光っているのに攻撃できない」という状態になる。
同じルールを2箇所に書くと、片方の更新漏れが必ずこの形で表面化する。
</details>

**問2.** `PersistentAura` の `ON_NEXT_SPELL` を消す処理を、
コスト計算より**前**に書くと何が起きるか。

<details><summary>答え</summary>

詠唱の宝珠の軽減が乗る前にオーラが消えるため、コストが下がらない。
「効果を消費する」処理は、その効果を使い終わった後に置く必要がある。
</details>

**問3.** 「相手の効果による破壊か」をターンプレイヤーで推定している。
この推定が破綻するのはどんなカードが登場したときか。

<details><summary>答え</summary>

相手のターン中に、自分の意思で自分のミニオンを破壊するカード
(例:「相手が攻撃したとき、自分のミニオン1体を破壊してもよい」)。
このとき破壊される側の持ち主はターンプレイヤーでないため、
自分の効果なのに「相手の効果」と誤判定される。
</details>
