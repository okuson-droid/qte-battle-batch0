# Batch 9 設計解説 — デッキビルダー: ファイルを「データベースの代わり」にする

## ⚡ 結論チートシート

- デッキは**JSONファイル**として出力し、対戦時に読み込む。**サーバは何も保存しない**(アカウント・DB不要)
- カードデータの正はサーバの台帳ひとつ。ビルダーは`/api/cards`から取得し、コピーを持たない
- **検証は二重に書く**。ビルダー側は操作の補助、対戦に使えるかの最終判定は`DeckValidator`(総合ルール第1章の実装)
- 効果が未実装のカード(未実装文明・未登録スペル)は**入口で弾く**。「入れられるのに何も起きない」が最悪だから
- デッキ未指定なら従来通りプリセット(おまかせ)で対戦できる

---

## 1. なぜアカウントとDBをやめたのが正解なのか ★

当初の相談は「アカウント登録してデッキを保存する」だったが、**ファイル出力方式に切り替えた判断は筋がいい**。得られるものと失うものを並べると理由が見える。

| | アカウント+DB方式 | ファイル方式(採用) |
|---|---|---|
| 必要な部品 | 認証・パスワード管理・DB・マイグレーション | なし |
| 継続コスト | 無料DBの期限切れリスク、スリープ | ゼロ |
| 責任 | 他人のパスワードを預かる | なし |
| デッキの持ち運び | ログインすればどこでも | ファイルを持ち歩く必要あり |
| 共有 | URLで共有しづらい | **ファイルを渡すだけで共有できる** |

失うのは「どの端末でもログインすれば同じデッキ」だけで、趣味の対戦なら十分実用になる。むしろ**デッキファイルを友達に送れば、そのままコピーして使える**——共有はこちらの方が簡単なくらいだ。

そしてこの選択は、Batch 0で決めた「サーバは状態を永続化しない」という方針の延長線上にきれいに乗る。**保存先がユーザーのPCになっただけで、サーバは相変わらず何も覚えない**。アーキテクチャに一貫性があると、後から足す機能も自然に収まる。

## 2. データの正はひとつ ★

デッキビルダーは大量のカードデータを必要とする。ここで安易な道は「ビルダー用にカードのJSONをもう1つ持つ」(元にした既存ツールがCSVを読み込んでいたのと同じ形)だが、それをやると**台帳を更新するたび2箇所を同期する羽目になり、必ずズレる**。

そこで`/api/cards`エンドポイントを立て、ビルダーは起動時にサーバから取得する。カードデータの実体はサーバ上の`qte-cards.json`ただ1つ。新しい文明を転記して台帳を更新すれば、ビルダーの選択肢は自動的に増える。**「同じ情報を2箇所に置かない」**は、DB設計の正規化と同じ原則がアプリ構成の階層で現れたもので、Batch 2の「動的ステータスを保存せず評価する」とも同じ発想である。

## 3. 検証を二重に書く理由 ★

デッキの規則(40枚・同名4枚・禁忌8枚・ハイランダー・文明制約)は、**ビルダーのJS**と**サーバの`DeckValidator`**の両方に書いてある。重複していて無駄に見えるが、役割がまったく違う。

- **ビルダー側**: 操作の補助。5枚目を追加させない、枚数バッジを赤から緑にする——「間違えにくくする」ためのもの
- **サーバ側**: 防壁。デッキファイルはユーザーのPCから来る、いくらでも書き換えられるテキストである。メモ帳で`"count": 99`に書き換えたファイルを読み込ませることは誰でもできる

**信用するのは自分の検証だけ**——WebSocketの操作検証(Batch 1)とまったく同じ構図で、入口が「メッセージ」から「ファイル」に変わっただけである。外部から来るデータはすべて信用しない、という原則の一貫した適用になっている。

## 4. 「何も起きないカード」を入口で弾く

`DeckValidator`は規則違反だけでなく、**効果が未実装のカードも拒否**する(未実装文明のカード全般と、効果が未登録のスペル)。理由はBatch 2で書いたのと同じ: 黙って何も起きないのは、気づけない最悪のバグだから。

デッキ構築の時点で弾けば、対戦中に「あれ? このスペル効かない」と悩む状況自体が発生しない。**エラーは発生源にできるだけ近い場所で検出する**。今は水・火の2文明だけが通るが、文明を実装するたびこの許可リストに足していけばよい。

## 5. UIの設計判断ふたつ

**リーダーを先に選ばせる。** メインデッキも禁忌デッキも「リーダーの文明」を基準に決まる(1-2, 1-3)ため、リーダーが決まらないとカードプールを絞れない。だからリーダー変更時はデッキをリセットする確認を出す——ルールの依存関係がUIの順序を決めている。

**カードプールを「メイン用/禁忌用」で切り替える。** 同じカードプール上でメインと禁忌を混在させると、クリックしたカードがどちらに入るのか分かりにくい。文明でタブを分けることで、「今どちらを組んでいるか」が常に明確になる。追加先の判定自体は文明の比較で自動的に決まるので、実装も単純なままである。

## ✅ 動作確認手順

1. 差し替えて起動 → ロビーに「デッキビルダー」リンクが出る
2. **構築**: リーダーを選ぶ → メイン用タブでカードをクリックして40枚、禁忌用タブで8枚 → バッジが両方緑になる
3. **出力**: 「デッキファイルを出力」→ `MyDeck.json`がダウンロードされる。中身をメモ帳で開くと、カードIDと枚数の素直なJSONになっている
4. **再編集**: 「デッキを読み込む」で先ほどのファイルを選ぶ → 続きから編集できる
5. **対戦**: ロビーの「デッキファイル」で選択 → 部屋を作成。対戦画面の自分側に「デッキ: MyDeck」と表示され、そのデッキで戦える
6. **検証の確認**: メモ帳でファイルの`count`を`99`に書き換えて読み込む → ロビーに赤いエラーで「同名カードは4枚までです」と出て、対戦に入れない
7. **おまかせ**: デッキファイルを選ばずに部屋を作れば、従来通りプリセットデッキで対戦できる

## ✅ 理解確認(2問)

1. デッキビルダーのJSにも枚数チェックがあるのに、サーバ側の`DeckValidator`が必要な理由を、具体的な攻撃手順を1つ挙げて説明せよ。
2. カードデータをビルダー用にもう1つのJSONとして持つ設計にした場合、どんな不具合が「いつか必ず」起きるか。
