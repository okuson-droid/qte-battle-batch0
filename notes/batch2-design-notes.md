# Batch 2 設計解説 — エフェクトシステム: 効果を「登録された部品」にする

## ⚡ 結論チートシート

- カード効果は`CardEffectRegistry`に「カードID → ラムダ」で登録する。**GameServiceはカードを1枚も知らない**
- 役割分担: GameService=いつ・誰が・何をしてよいか / Registry=何が起きるか / GameActions=共通の基本操作 / StatCalculator=動的ステータスの評価
- ON_SUMMON(召喚時)とON_ENTER(登場時)は別トリガーとして発火する。召喚は両方、効果で「出す」はON_ENTERのみ(確定裁定)
- 攻撃力は保存せず**評価のたびに計算**する。「手札枚数に連動」をキャッシュなしで正しく保つため
- 【還元】と【知識】はキーワードなので、個別カードの登録ではなく共通処理(GameActions/Registry)が一律に扱う
- 動作確認: スペルがプレイできる。知識の守護者の攻撃力がドローで増える。流転の書がマナに還る

---

## 1. なぜ「if文の羅列」にしないのか ★

一番素朴な実装はGameServiceにこう書くことだ。

```java
if (cardId.equals("QTE-0042")) { draw(2); }
else if (cardId.equals("QTE-0029")) { if (hand >= 5) heal(3); }
else if ...  // カードが増えるたびここが伸びる
```

これはカード10枚なら動く。しかし全文明150枚超が入るとGameServiceは千行級の分岐になり、**「ルールの進行」と「個々のカードの中身」が絡まって**どちらも読めなくなる。修正のたびに巨大ファイルを触るので、ある文明の修正が別の文明を壊す事故も起きる。

そこで責務を切る。GameServiceが知っているのは「スペルを唱えたら効果を解決し、そのあと墓地(または還元でマナ)へ送る」という**手続きの骨格だけ**。効果の中身は`CardEffectRegistry`から引く。

```java
effects.resolveSpell(master.id(), contextOf(...));   // 中身は知らない
actions.disposeUsedSpell(room, player, master);       // 後処理の骨格だけ知っている
```

この構造の価値は**変更の局所化**にある。今後カードを追加する作業は「Registryに数行登録する」だけになり、GameServiceは原則もう変更しない。EM-Springで学んだ言葉でいえば、これは制御の流れ(フレームワーク側)と個別処理(アプリ側)を分けるDIコンテナの発想の縮小版で、Registryは「カード効果のコンテナ」である。

## 2. トリガーの分類 — ON_SUMMONとON_ENTERは別物 ★

仕様確定フェーズで得た裁定「召喚と『出す』は区別され、【召喚時】は召喚のみ、【知識】は登場すべてで発動」を、コードでは`TriggerType`の2値としてそのまま表現した。召喚処理はこう発火する。

```java
effects.fire(TriggerType.ON_SUMMON, minion, ctx);  // 【召喚時】
effects.fire(TriggerType.ON_ENTER,  minion, ctx);  // 登場時(知識など)
```

将来、黄泉還る水龍のような「効果で出す」を実装するときは**ON_ENTERだけを発火する**。ルール上の区別が、発火するトリガーの違いという1点に集約されているのがこの設計の要点である。ルールの語彙(召喚/出す/登場時)とコードの語彙(ON_SUMMON/ON_ENTER)が1対1で対応しているので、新しいカードの裁定に迷ったらルール文書を読めばコードの書き方が決まる。

【知識】だけはRegistryの`fire()`内の共通処理にした。知識は個々のカードの効果ではなくキーワードであり、「知識を持つカードすべてに同じ登録を50回書く」のは重複だからである。Batch 1でGameServiceに直書きしたad hoc実装は、ここに移設されて消えた——**動くものを先に作り、仕組みができたら吸収する**というリファクタリングの実例として、Batch 1のコードと見比べてみてほしい。

## 3. 動的ステータス — 保存せず、評価する ★

知識の守護者(攻撃力=手札の枚数)と溢れ出る英知(ターン中、手札の枚数分+1、**常に連動**)が今回の主役だ。素朴には「効果解決時に攻撃力を書き換える」実装が浮かぶが、それではドローや手札消費のたびに書き直しが要り、1箇所でも漏れると**画面と実際の値がズレる**。この種の更新漏れバグは再現条件が複雑で、発見が非常に遅れる。

採用したのは逆の方針で、攻撃力をどこにも保存しない。`StatCalculator`が**評価のたびに**「基礎値 → 動的SET → 静的SET → 動的ADD → 静的ADD」を計算する。溢れ出る英知は「+N」という数値ではなく「このオーラが有効」という**事実だけ**を`PlayerState.thisTurnAuras`に記録し、Nは評価時に手札を数えて決める。参照元(手札)が変われば、次の評価が勝手に正しい値を返す。更新処理が存在しないので、更新漏れも存在しえない。

これはDB設計で学んだ正規化と同じ思想である。導出できる値を保存するのは冗長であり、冗長は不整合の温床になる。計算コスト(数体のミニオン×数個の修正)は無視できるので、正しさを取る。

## 4. GameActionsという共通の土台

「ドローする」はルール本体(ターン開始時)もカード効果(スプラッシュ・ドロー)も使う。効果側がGameServiceを呼ぶと循環参照になるため、基本操作を`GameActions`に切り出し、両者がそこに依存する形にした(依存の向きを一方通行に保つ)。

副産物として、**山札切れ敗北の判定が1箇所になった**。ドローの経路が何本あっても(ターン開始・知識・スペル)、全部が同じ`drawCards`を通るので、敗北判定を書き忘れた経路が存在しえない。【還元】も同様で、スペル使用後とミニオン破壊後という2つの経路が同じ`sendToTrashOrRestore`に合流するため、「墓地の代わりにマナへ」の判断は1箇所にしかない。

## 5. 未実装スペルは「黙って何もしない」より「拒否」

Registryに登録のないスペルをプレイしようとすると例外で拒否される。コストだけ払って何も起きないのが最悪のバグ(気づけないバグ)だからで、「未実装です」と言って止まるほうが遥かにましである。データ(台帳JSON)とコード(Registry)の突き合わせで実装漏れを機械的に検出できる構造にもなっている。

## 6. デッキ更新と、意図的に「まだ無い」もの

デッキはミニオン30枚+スペル10枚に更新した。Batch 3に残っているのは: **対象指定**(手札を喰らう大蟹・双流の幻術師——「どれを選ぶか」のUIとサーバ検証)、**ウェポンとリーダー攻撃**(三叉槍・氷結の杖と凍結デバフ)、**特殊召喚**(プレサージュ・双翼・ポセイドン系)、**墓地トリガー**(黄泉還る水龍——ゾーン横断監視)、**リーダー起動能力**、**ピュア・エレメント**、マリガン。

仮ルールを1つ追加した: **リーダーの回復は上限20(初期値)まで**。総合ルールに規定がないため要確認。

## ✅ 動作確認手順

1. 差し替え(src消去→展開→F5→Maven Update→Clean)して起動、2タブで対戦開始
2. **スペル**: メインフェイズにアクア・サーチ等をクリック → ドローされ、カードが墓地へ(墓地カウント増)
3. **還元**: 流転の書か知識の守り手(破壊されたとき)→ 墓地でなく**マナが1枚増える**(裏向きチップ)
4. **動的SET**: 知識の守護者を出す → 攻撃力が手札枚数と一致。ドローすると表示が増える
5. **動的ADD**: 溢れ出る英知 → 自分の全ミニオンの攻撃力が手札分上がり、カードを使うたび下がり、ターン終了で消える
6. **ON_SUMMON**: 水鏡の幻術師で2ドロー。潮流の魔導士は手札5枚以上のときだけ回復
7. **サブフェイズ**: バトルフェイズの後でもスペルだけは使える(ミニオンは拒否される)
8. **未実装の拒否**: ピュア・エレメントをプレイ → 「Batch 3で対応」のエラーが出る

## ✅ 理解確認(2問)

1. 溢れ出る英知の「+N」を、効果解決時に各ミニオンへ`StatModifier(ADD, N, THIS_TURN)`として積む実装にした場合、どんな操作の後に仕様(常に連動)と食い違うか。
2. 新カード「コスト3のスペル: 自分のリーダーを2回復し、2枚引く」を追加するとき、変更が必要なファイルはどれか(答えは1つ)。
