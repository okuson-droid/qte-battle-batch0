## ⚡ 結論チートシート

1. 風文明が参照するのは資源の「量」ではなく行動の「履歴」である。だから a1 が要る。
2. 風文明は選択のタイミングを「使用時」から「解決の途中」へ動かした。だから a9 が要る。
3. **a9 は投資である。** 導入と引き換えに、暫定策だった `AutoChoice` の適用範囲が減る方向に動き出した。

## ★★★ 1. a1 — 使用カウンタと ON_CARD_USED

### 1-1. 何が無かったか

`PlayerState` が持っていたのは `playedCardThisTurn`(真偽値)だけだった。
「使ったかどうか」は分かるが「何回使ったか」は分からない。風文明の9枚がこれを要求する。

```java
private int cardsUsedThisTurn = 0;    // 使用したカードの総数(起動能力の発動を含む)
private int spellsCastThisTurn = 0;   // うちスペルの回数(詠唱の疾風騎士)
```

`playedCardThisTurn` は海皇ポセイドンが「メインフェーズ開始時か」の近似として使っているため、
意味が違う。統合せずに並べて持つ。

### 1-2. ★最重要: 加算は「効果解決の後」

裁定1により、使用カウンタは**すべて自身を含まない**。実装上はこれが加算位置の指定になる。

```
検証 → コスト計算 → 支払い → 効果解決 → ★カウンタ加算 + ON_CARD_USED発火★
```

この位置に置くと、カウンタを読むすべての効果が自動的に
「自分より前に使ったカードの枚数」を見ることになる。カードごとに `-1` する補正は要らない。

| カード | この位置がもたらす帰結 |
|---|---|
| 詠唱の風詠士(L010) | コスト評価時に `cardsUsedThisTurn == 2` なら軽減 = そのカードが3枚目 |
| 神風の大号令(0113) | 3枚目に唱えると解決時 `== 2` → 全体+2 |
| ガイル・フォックス(0117) | 【召喚時】は解決中に発火するため自身を含まない |
| 突風のまとめ役(0134) | 召喚した瞬間には自分にバフが乗らない |

### 1-3. 設計書からの変更点(承認済み)

設計書は挿入箇所を `playMinion` / `playSpell` / `playWeapon` の3箇所としていたが、
実装では **`playCard` の `switch` 直後の1箇所**に集約した。
3種類とも `playCard` を通り、`switch` を抜けた時点でどれも解決が終わっているため意味は同じで、
かつ**入れ忘れの事故が構造的に起きなくなる**。

`playCard` を通らない経路には個別に入れた。結果として挿入箇所は次の5つになる。

```
playCard(switch直後) / playTabooCard / specialSummon / summonFromGrave
/ useLeaderAbility / useMinionAbility
```

**起動能力の発動も数える**(発注者確認済みの横断ルール。project-reference 2-9)。
リーダーだけでなくミニオンの起動能力(a6)も対象である。

### 1-4. ★なぜカウンタを読むだけでは足りないのか

突風のまとめ役と暴風の双剣は「使用する**たび**」であり、これはカウンタの参照では表現できない。
`TriggerType.ON_CARD_USED` を新設した理由が2つある。

- **遡及の問題。** まとめ役は自分以外の全ミニオンにもバフを配る。カウンタの差分で計算すると、
  バフが配られた**後に**場に出てきたミニオンにも遡って乗ってしまう
- **発生源の寿命の問題。** まとめ役が破壊されても、すでに配ったバフはターン終了時まで残るはずである。
  カウンタ参照だと発生源が消えた瞬間にバフも消える

そこで「その瞬間に場にいるミニオンへ `StatModifier(ATTACK, ADD, 1, THIS_TURN)` を積む」形にした。
**期限の管理は既存の修正スタックがそのまま担う**ため、追加したコードは発火経路だけである。

> **採用しなかった案:** ミニオンに登場時のカウンタ値を記録し、現在値との差を攻撃力に足す方式。
> フィールド1本で済み `ON_CARD_USED` を新設せずに済むが、上記2つの意味論を再現できない。
> 風文明単体では差が出にくいが、風の全体バフと闇の全体破壊が同居する禁忌デッキで必ず食い違う。

### 1-5. ウェポンだけ置き場所が違う

暴風の双剣はウェポンであり `MinionInstance` を持たないため、修正を積む先がない。

```java
// PlayerState
private int weaponAttackBonusThisTurn = 0;
```

`StatCalculator.effectiveWeaponAttack` で加算し、次の2箇所で0に戻す。

- `GameActions.onWeaponLeftPlay`(ウェポンが場を離れたとき)
- `GameService.finishEndTurnCleanup`(ターン終了時)

**☆補足:** ターン終了時のクリアを `startTurnReset` ではなく `endTurn` に置いたのは意図的である。
`startTurnReset` は自分のターンが始まるときにしか走らないため、そこでクリアすると
**相手のターンの間ずっと加算が残ったまま相手に見えてしまう**。

## ★★★★ 2. a9 — 割り込み選択システム(本バッチの中心)

### 2-1. 何が問題だったか

これまでの対象選択は **「カードを使う瞬間に、必要な選択をすべて終える」** 前提でできていた。

```
validateTargets → 支払い → removePlayedAndTargets → resolveSpell
```

この一直線の経路には、**解決の途中でクライアントに問い合わせて再開する場所がない**。
闇文明ではこれを `AutoChoice` による自動決定で回避した(設計判断32)。
風文明の裁定5・6が「本人に選ばせる」ことを要求したため、回避しきれなくなった。

唯一の例外が 11b の降臨の伝道師(`pendingReveal`)だったが、これは**そのカード1枚専用の作り**だった。

### 2-2. ★設計: 新しく作らず、既にあるものを一般化する

`pendingReveal` は廃止し、降臨の伝道師も新しい器に載せ替えた。
そのうえで、旧実装が1つのリストに兼ねさせていた**2つの役割を分離した**。ここが本項の要点である。

| 分離後 | 役割 | 型 |
|---|---|---|
| `PlayerState.revealedZone` | 公開されたカードの**置き場**(一時公開領域) | `List<String>` |
| `PlayerState.pendingChoice` | プレイヤーへの**問い合わせ** | `PendingChoice` |

旧 `pendingReveal` は「公開中のカード」と「選択待ちであること」を同じリストで表していた。
そのため、**公開を伴わない選択**(手札から捨てる・墓地から回収する)に流用できなかった。
分離したことで、`pendingChoice` は公開領域と無関係に単独で使えるようになっている。

```java
public record PendingChoice(
        Kind kind, List<String> candidates, int min, int max,
        ResumePoint resumeAt, String prompt) { }
```

### 2-3. ★再開先を「値」として持つ

Java には効果の途中で処理を中断し、後から同じ場所へ戻る仕組み(継続)がない。
したがって「どこから再開するか」を値として持たせるしかない。選択肢は2つあった。

- **A案: 継続を `Consumer<EffectContext>` として保持する。** 書き味は最も良い。しかし
  `GameState` はビュー生成のたびに走査される可変データであり、**シリアライズできない関数**を
  紛れ込ませると、将来「状態のスナップショットを取る」「リプレイを保存する」といった拡張が全部塞がる
- **B案(採用): 再開先を列挙体で持ち、`CardEffectRegistry` 側で `switch` する。**
  カードが増えるたびに列挙体が1つ増えるが、状態は文字列と数値だけで表現できる。
  `TriggerType` や `PersistentAura.Expiry` と同じ流儀であり、既存の設計と揃う

```java
public enum ResumePoint {
    MISSIONARY_SUMMON,     // 降臨の伝道師(既存 pendingReveal からの移行。12aで実装済み)
    TAILWIND_DISCARD,      // 選択の追い風        ┐
    MANA_CONVERT_PUT,      // 風のマナ変換        │
    WINDHOLE_SECOND,       // 回帰の風穴          ├ 12b でカード効果と同時に実装
    GUARD_STAFF_TARGET,    // 風護の杖            │
    GALE_KNIGHT_RECOVER    // 詠唱の疾風騎士      ┘
}
```

### 2-4. 候補の渡し方

**クライアントは「候補の並び順の位置」を送り返す。** 手札の位置や `instanceId` を直接送らせない。
中断している間に盤面が変化しても、位置の解釈がずれないようにするためである。

候補には**選べるものだけ**を入れる。降臨の伝道師なら【守護】を持つカードの位置だけが候補になり、
公開された4枚全部は `revealedZone` 側に入って画面に表示される。
「見えているもの」と「選べるもの」を別々に持てるのは、2-2 で役割を分離したおかげである。

### 2-5. 中断中は他の操作を全部塞ぐ

```java
private void requireTurnPlayer(GameState state, String playerId) {
    if (!playerId.equals(state.getTurnPlayerId())) { ... }
    if (state.playerOf(playerId).getPendingChoice() != null) {
        throw new IllegalStateException("先に選択を解決してください");
    }
}
```

ターンプレイヤー判定の直後に置くことで、**多数ある呼び出し元を個別に触らずに済む**。
`resolveChoice` 自身はこのメソッドを経由しないため、選択操作そのものは塞がれない
(11b の `resolveRevealChoice` が `requireTurnPlayer` を意図的に迂回していたのと同じ理由である)。

### 2-6. ★★ ターンエンド中の割り込み — `endTurn` の分割

**a9 で唯一、構造的な改修を伴ったのがここである。**

改修前の `endTurn` は、後始末をして末尾で `beginTurn` を直接呼び、
ターンの受け渡しまでを1つのメソッドで完了させていた。
ターンエンド時効果(詠唱の疾風騎士)が選択待ちになると、
**選択の解決を待たずに相手のターンが始まってしまう。**

```java
public void endTurn(...)                  // ON_TURN_END発火 → 後始末 → advanceTurn
private void finishEndTurnCleanup(...)    // 「このターン中」の効果の掃除
private void advanceTurn(...)             // beginTurn を呼ぶ
private void advanceTurnIfPending(...)    // 選択解決後に呼ばれ、保留していた受け渡しを行う
```

選択が保留された場合は `GameState.turnHandoffPending` を立てて戻り、
`resolveChoice` の末尾から `advanceTurnIfPending` が続きを実行する。

**「ターンの終了」と「次のターンの開始」を別の出来事として分ける**改修であり、
総合ルール上もこの2つは別の事象であるため、構造としてはむしろ正しくなった。

☆ `advanceTurnIfPending` は「まだ別の選択待ちが残っていれば何もしない」と書いてある。
将来、1回のターンエンドで複数の割り込みが連鎖しても壊れない。

### 2-7. AutoChoice への影響

a9 の導入により、詠唱の疾風騎士のスペル回収と風護の杖の付与先は
**新規に `AutoChoice` を足さずに済むようになった**。

一方、**既存の闇文明5枚**(冥界神ハデス・死者蘇生・執念の暗殺者・不滅のネクロマンサー・死霊の収鎌)
**は本バッチでは移行していない。** 5枚の効果を同時に書き換える作業であり、
風の実装と混ぜると事故が起きたときにどちらが原因か切り分けられなくなるためである。
移行は独立したバッチで行う。

## ★★★ 3. a2 — 攻撃回数の属性化(設計判断7の回収)

### 3-1. 直書きを動的評価に変える

改修前は2箇所にハードコードされていた。

```java
if (attacker.getAttacksUsedThisTurn() >= 1) { ... }        // RuleGuards
if (player.isLeaderAttackedThisTurn()) { ... }             // GameService
```

設計判断7は「攻撃回数は固定1ではなくカードごとの属性として持つ」と宣言済みだったが、
連撃の巨岩(土)が未実装のため据え置かれていた。風が3枚同時に要求するため、ここで回収した。

```java
public int maxAttacks(GameState state, PlayerState owner, MinionInstance minion) {
    int max = 1;
    // 印刷された「2回攻撃」を持つカードは 12b でここに追加(サイクロン・フェンサー)
    for (StatModifier m : minion.getModifiers()) {
        if (m.stat() == StatModifier.Stat.EXTRA_ATTACKS) max += m.value();
    }
    return Math.max(1, max);
}
```

`StatModifier.Stat` に `EXTRA_ATTACKS` を追加した。ツイン・ストライクは
`EXTRA_ATTACKS, ADD, 1, THIS_TURN` を積むだけで済み、
**期限の管理は既存の `expireThisTurnModifiers` がそのまま担当する**。
専用フィールドを足さないのはこのためである。

### 3-2. ☆リーダー側は判定層への移設を伴った

`leaderAttackedThisTurn`(boolean)を `leaderAttacksUsedThisTurn`(int)に置き換えた。
併せて、`GameService.leaderAttack` に直書きされていた3判定
(未装備・攻撃済み・凍結)を `RuleGuards.leaderAttackDenial` へ移した。

これらはミニオン側が 11a で判定層へ移されたときに**取り残されていた**ものであり、
設計判断34の趣旨に反していた。移設によって、ビュー(ボタンの活性判定)が
検証とまったく同じ判定を呼べるようになる。

```java
// GameViewBuilder。判定の重複記述が消え、表示と検証のズレが構造的に起きなくなった
boolean leaderCanAttack = attackerSide && guards.leaderAttackDenial(state, player) == null;
```

## ★★ 4. a3 — 体力修正

攻撃力を上げるカードは水・火・闇に多数あったが、**体力そのものを上げるカードは風が初めて**である。
改修前の体力は `master.hp() - damage` の固定計算だった。

```java
public int getMaxHp() { /* SET を適用してから ADD を適用する(攻撃力と同じ順序) */ }
public int getCurrentHp() { return getMaxHp() - damage; }
```

### ★なぜ `StatCalculator` ではなく `MinionInstance` に置いたのか

攻撃力とコストを `StatCalculator` に出したのは、それが**盤面を参照する動的な値**
(手札枚数・墓地枚数)だったからである。風の体力修正はすべて固定値(+1 / +2)であり、盤面を見ない。

`MinionInstance` 内で閉じれば、`getCurrentHp()` の呼び出し元
(破壊判定・ビュー・`HP_5_OR_LESS` フィルタ)を**1箇所も書き換えずに済む**。
盤面参照型の体力修正が出た時点で `StatCalculator` へ移す。

☆ 設計書は「`heal()` の上限も `getMaxHp()` に変える」としていたが、**変更は不要だった**。
現行の `heal()` は `damage = max(0, damage - amount)` と、受けているダメージを減らす形で
書かれており、最大体力を参照していない。構造上、超過回復は起こりえない。

ビューは `master.hp()` を送っていたため `minion.getMaxHp()` に変えた。
これをしないと「+2された」ことがプレイヤーに見えない。

## ★★ 5. a4 — ON_TURN_END の発火

`TriggerType.ON_TURN_END` は Batch 2 で宣言されていたが、**発火する場所がどこにもなかった**。
`GameService.endTurn` に「該当カード実装時にここで発火する」というコメントだけが残っていた。

発火位置には条件がある。

- **後始末より「前」に置く。** 「このターン5回以上スペルを撃っていたら」を条件にするため、
  カウンタと `THIS_TURN` の修正が**生きているうちに**評価しなければならない
- 対象は両プレイヤーの場のミニオン。ターンプレイヤー側から先に処理する
- 発火の途中で決着した場合(山札切れ)は打ち切る

墓地からのスペル回収は a9 の割り込みで本人に選ばせるため、この経路が 2-6 の `endTurn` 分割を要求した。

## ★★ 6. a5 — 強化使用と行き先の置換

### 6-1. 既存の「してもよい」と何が違うか

これまでの任意効果はすべて対象選択の任意化(`Requirement.optional`)で表現されていた。
つまり「選ぶか選ばないか」であり、選ばなくても**支払う額は変わらない**。
風の2枚は対象を伴わずに支払う額そのものが変わる**二者択一**であり、既存の枠組みでは表現できない。

```java
public record EnhancedCostSpec(int extraCost, String prompt) {}
```

- 通信: `PlayCardRequest` に `boolean enhanced` を追加
- コスト: `GameService` が `effectiveCost + extraCost` で支払わせる
- 効果: `EffectContext` に `enhanced` を持たせ、効果側が分岐する

### ★なぜ a9 に載せないのか

コストに影響する選択は**支払いより前に確定していなければならない**。
a9 は「効果の解決の途中で問い合わせる」仕組みであり、そのときには支払いはとっくに終わっている。
性質が違うものを同じ仕組みに載せると、どちらの意味論も壊れる。

### 6-2. 行き先の置換

スペルは解決後に必ず `actions.disposeUsedSpell`(墓地 or 還元)を通っていた。風の2枚はここを置き換える。

```java
@Setter private SpellDisposition pendingSpellDisposition; // TO_HAND / TO_DECK_BOTTOM / null
```

効果が書き込み、`playSpell` が読んで消費する。
これは `pendingFireMinionDiscount`(剛火の将)・`pendingSacrificeCount`(死者蘇生)と同じ
**「効果と呼び出し元のあいだで値を1個受け渡す」既存の型**であり、新しい流儀を持ち込んでいない。

**禁忌由来のスペルは置換を受けない**(発注者確認済み)。総合ルール3-6により、
禁忌カードは使用され終わると消滅ゾーンへ行くことが確定しているためである。
効果が誤って書き込んでも通常プレイに漏れないよう、`playTabooCard` の前後でクリアしている。

> **採用しなかった案:** `TargetSpec.Kind` に `MODE` を足して選択肢そのものを対象として扱う。
> 汎用性は高いが、`Kind` は「盤面のどれを指すか」を表す語彙であり、モード選択は指す先を持たない。
> 意味の違うものを同じ列挙体に入れると、以降 `validateTargets` の `switch` が
> 対象検証とモード検証の2種類を抱えることになる。

## ★★ 7. a7 — 合計指定 / ★ a8 — 手札から裏向きマナへ

### 7-1. a7: `Kind` を増やさずに解く

「場か手札のカードを2枚」は、**1件の要求のなかで2つのゾーンから選ばせる**初の例である。

```java
public record TargetSpec(List<Requirement> requirements, int combinedTotal) { ... }
```

各要求は `upTo` で 0〜2 枚を受け付け、`validateTargets` の最後に合計だけを見る。
**`Kind` も `TargetChoice` も変えずに済む。**

> **採用しなかった案:** `Kind.FIELD_OR_HAND` を足す。引き継ぎ書は「新しい選ぶものが出たら
> `Kind` を1つ増やすほうが影響範囲が小さい」としており一見方針に沿うが、
> これは**新しい選ぶものではなく、既存の2つを同時に選ばせる話**である。
> `Kind` を足すと `TargetChoice` が「手札インデックスとミニオンIDの混在」を運ぶ必要が生じ、
> 検証・UI・DTO のすべてに分岐が増える。`Kind` の増設が正しいのは
> 「今まで指せなかったものを指す」場合に限られる(`WEAPON` がそれだった)。

### 7-2. a8: 風から闇の資源が生まれる

マナ関連の操作は闇文明でほぼ出揃っていたが、**手札から直接マナへ置く**操作だけがなかった。

```java
public boolean putHandCardIntoManaFaceDown(GameRoom room, PlayerState owner, int handIndex)
```

マナチャージ(1ターン1回)とは別処理であり、`manaChargedThisTurn` は立てない。

☆ 裏向きマナは闇文明の参照資源(禁忌の代償・不滅のネクロマンサー)であり、
これは**風から闇の資源を生成する初めての経路**になる。6文明そろった時点の整合チェックで見る。

## ★★★ 8. a6 — ミニオンの起動能力

**1枚(静空の風使い)のために3つの新概念が必要になった。**

### 8-1. ミニオンのタップ状態

`MinionInstance` にタップの概念がなかった。攻撃は `attacksUsedThisTurn` で数えており、
タップを使っていなかったためである。

総合ルール6章-2は「自分の場・マナゾーンの全タップ状態カード」をアンタップすると定めており、
**場のカードにタップ状態が無かったこれまでの実装のほうがルールから逸脱していた。**

裁定4により、タップ中のミニオンは攻撃できない(判定は `RuleGuards` に1件追加)。
一方**【守護】【潜伏】はタップ中も機能する**ため、キーワードの評価には一切手を入れていない。

☆ 攻撃回数を `tapped` ではなく `attacksUsedThisTurn` で数え続けているのは、
「1ターンに2回攻撃できる」カード(a2)があるためである。両者は別の概念として共存する。

### 8-2. ★`LeaderAbilitySpec` と統合しない

構造はほぼ同じだが、**共通なのは名前だけ**である。

| | リーダー | ミニオン |
|---|---|---|
| コスト体系 | MP | 自身をタップ |
| 使用制限 | 1ターン1回のフラグ | タップ状態そのもの |
| 呼び出し経路 | プレイヤー単位 | インスタンス単位 |

サービスクラスの分割基準と同じ理屈である。
共通化すると、3つの違いを吸収するための分岐が仕様型の中に生まれる。

### 8-3. マナのアンタップ

`GameActions` にはタップ(支払い)と表裏の反転はあったが、**アンタップする操作がなかった**。
マナ加速が既存カードプールに存在しなかったためである。`untapMana` を追加した。

## ✅ 理解確認

<details>
<summary>Q1. 使用カウンタの加算を「効果解決の後」に置くと、なぜカードごとの補正が不要になるのか。</summary>

解決の途中でカウンタを読む効果からは、常に「自分より前に使ったカードの枚数」しか見えないためである。
自分自身がまだ数えられていないので、`-1` の補正を各カードに書く必要がない。

もし加算を「支払いの直後」に置くと、自分を含んだ値が見えてしまい、
参照するカード9枚すべてに「自分を除く」補正を書くことになる。
補正の書き忘れは1枚単位で起き、テストでも見つけにくい。
**位置の選択によって、9箇所の間違えうるコードが0箇所になっている。**
</details>

<details>
<summary>Q2. 突風のまとめ役を「カウンタの差分」で実装すると、具体的にどう壊れるか。</summary>

2つ壊れる。

1. **遡及。** カウンタが3のときにミニオンAが場に出て、その後4になったとする。
   差分方式ではAの攻撃力に「4-(登場時の値)」が乗るが、Aが登場する前の使用でも
   バフが乗ってしまう組み合わせが生まれる。本来「使用のたびに、その瞬間いるミニオンに配る」効果である
2. **発生源の寿命。** まとめ役が破壊されると、差分の計算元(まとめ役がいること)が消えるため
   バフも消える。しかし効果は「ターン終了時まで」であり、発生源が消えてもバフは残るはずである

イベントで配る方式なら、配った時点で `StatModifier` がミニオン側に積まれるため、
どちらも自然に正しくなる。
</details>

<details>
<summary>Q3. a9 で「継続をラムダで持つ」案を捨てた理由を、GameState の性質から説明せよ。</summary>

`GameState` はビュー生成のたびに走査される可変データであり、
将来「状態のスナップショットを取る」「リプレイを保存する」といった拡張の対象になりうる。
そこにシリアライズできない関数(ラムダ)が紛れ込むと、それらの拡張が**すべて塞がれる**。

列挙体 + `switch` なら状態は文字列と数値だけで表現でき、いつでも保存・復元できる。
書き味は落ちるが、失う自由度のほうが大きい、という判断である。
</details>

<details>
<summary>Q4. 旧 pendingReveal が持っていた「2つの役割」とは何か。分離して何ができるようになったか。</summary>

「公開されたカードの置き場」と「プレイヤーが選択待ちであること」の2つを、
1本の `List<String>` が兼ねていた。「リストが空でなければ選択待ち」という表現である。

分離したことで、**公開を伴わない選択**が表現できるようになった。
選択の追い風(手札から捨てる)・詠唱の疾風騎士(墓地から回収)は何も公開しないため、
旧構造では `pendingReveal` に載せられなかった。

また逆に、「4枚公開されているが、選べるのはそのうち2枚だけ」という状態も自然に表せる
(`revealedZone` に4枚、`candidates` に2枚)。
</details>

<details>
<summary>Q5. endTurn の分割は、なぜ「構造としてむしろ正しくなった」と言えるのか。</summary>

総合ルール上、「ターンの終了」と「次のターンの開始」は別の事象だからである。
改修前のコードは、実装の都合でこの2つを1つのメソッドに詰め込んでいた。

割り込みという新しい要求が来たとき、その要求は
「2つの事象のあいだに、第3の事象(選択の解決)が挟まりうる」というものだった。
2つが分かれていなければ挟めない。**要求のほうが、元の構造の誤りを暴いた**形になっている。
</details>

<details>
<summary>Q6. a3 の体力修正を StatCalculator ではなく MinionInstance に置いた判断基準は何か。その基準が変わるのはどんなときか。</summary>

基準は「**その値の計算に盤面の参照が要るか**」である。
攻撃力とコストは手札枚数や墓地枚数を見るため、盤面全体にアクセスできる `StatCalculator` に置いた。
風の体力修正はすべて固定値(+1 / +2)であり、ミニオン自身の修正スタックだけで計算できる。

`MinionInstance` 内で閉じれば `getCurrentHp()` の呼び出し元を1箇所も書き換えずに済む
(破壊判定・ビュー・`HP_5_OR_LESS` フィルタが該当する)。

基準が変わるのは「自分の場の【守護】の数だけ体力+1」のような
**盤面参照型の体力修正**が登場したときである。そのとき `StatCalculator` へ移す。
</details>

<details>
<summary>Q7. a7 で Kind を増やさなかったのに、過去に WEAPON という Kind は増やしている。この違いは何か。</summary>

`Kind` は「**盤面のどれを指すか**」を表す語彙である。

- `WEAPON` は「今まで指せなかったもの(装備中のウェポン)を指せるようにする」変更だった。
  語彙が足りていなかったので足すのが正しい
- a7 が要求したのは「**既存の2つ(手札と場)を同時に選ばせる**」ことであり、
  新しく指す先が増えたわけではない。ここで `Kind` を足すと `TargetChoice` が
  「手札インデックスとミニオンIDの混在」を運ぶ必要が生じ、検証・UI・DTO すべてに分岐が増える

**語彙を増やすのか、語彙の組み合わせ方を増やすのか**という違いである。
</details>
