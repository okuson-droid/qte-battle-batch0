Batch 10a で作った土台の上に、闇文明のカード28枚を載せる。
これで水・火・闇の3文明84枚が対戦で使える状態になる。

## 1. 対象選択に「墓地」という場所が増えた ★

これまで対象に取れるのは **手札・場のミニオン・マナゾーン** の3つだった。
闇文明は墓地を資源として扱うため、4つ目が必要になる。

```java
public enum Kind {
    HAND, MINION, MANA,
    TRASH   // ← 追加
}
```

実装で一番迷うのは「選んだカードを墓地から取り除くのは誰か」である。
手札の対象カードは、検証の時点で**手札から取り除かれた状態**で効果に渡される
(そういう仕様になっている)。同じ扱いにすると、墓地の場合は困る。

- 墓場の怨念集合体 → 手札に加える
- 禁忌の墓地利用 → マナゾーンに置く
- (蘇生系) → 場に出す

**行き先がカードごとに違う。** そこでマナゾーンと同じ方式を採り、
墓地のカードは **墓地に残したままIDだけを効果に渡す** ことにした。
移動は効果自身が `returnFromTrashToHand` / `putTrashCardIntoManaFaceDown` などで行う。

```java
public record Selection(List<String> handCardIds, List<TargetedMinion> minions,
        List<ManaCard> mana, List<String> trashCardIds) {
```

### 「好きな数」を表現する upTo

死者蘇生の「好きな数の自分のミニオンを破壊してもよい」は、
既存の `count`(ちょうどN個)でも `optional`(0個かN個)でも表せない。
そこで **0個からcount個までのどこでもよい** ことを示す `upTo` を足した。

```java
public static Requirement upTo(Kind kind, Side side, int max, String prompt, Filter... filters)
```

クライアント側には「この内容で確定」ボタンが出る。
禁忌の墓地利用の「墓地に1枚しかなければ1枚だけ置く」も、同じ仕組みで表現できた。

## 2. 「代償を払えないなら使えない」をどこで判定するか ★

闇には代償を伴うカードが多い。

- 禁忌の代償: 裏向きマナ1枚を破壊する
- 絶望の連鎖: 自分のミニオン1体を破壊する
- マナを貪る怨霊: 表向きのマナを裏向きにする

発注者確認済みのルールでは、**代償を払えない場合は使用できない**。
問題は、この判定を「どのタイミングで」行うかである。

効果の中で判定して例外を投げるのは**間違い**である。
効果が走る時点では既にコストを支払い、カードは手札から消えている。
そこで例外を投げると、MPだけ失ってカードが宙に浮く。

```java
public void playCard(...) {
    CardMaster master = cards.findById(peekHand(player, handIndex));
    effects.requirePlayable(master.id(), state, player);   // ← 支払いより前
    switch (master.type()) { ... }
}
```

「検証(状態を変えない)→ 支払い → 手札除去 → 効果」という既存の順序に、
検証をもう1段追加した形である。**状態を変える前にすべての拒否を済ませる**
という原則は、Batch 3 から一貫して守られている。

リーダー起動能力にも同じ問題があるため、`LeaderAbilitySpec` に
`condition` を足した(冥府の禁皇は裏向きマナが無ければ使えない)。
ビューの `usable` も同じ条件を見るので、押せないボタンは最初から無効化される。

## 3. AutoChoice — 「今は解けない問題」の隔離 ★★

闇の4枚は、効果の**解決中**に選択を要求する。

| カード | 解決中に必要な選択 |
|---|---|
| 冥界神ハデス | 全体破壊した**後**に、どの体を蘇生するか |
| 死者蘇生 | 生贄を破壊した**後**に、何を蘇生するか |
| 執念の暗殺者 | 破壊が起きた瞬間に「引いてもよい」の可否 |
| 不滅のネクロマンサー | 同じく「蘇生してもよい」の可否 |
| 死霊の収鎌 | 攻撃の最中に墓地から1枚 |

現在のアーキテクチャは「使う瞬間に全部選び終える」前提であり、
解決を中断して問い合わせる経路が無い。これを正しく解くには
**割り込み選択システム**(保留中の選択と継続の保存、WebSocketでの往復、
再開時の状態再検証)が必要で、解決経路全体に影響が及ぶ。

そこで暫定方針として自動解決を選んだ(発注者確認済み)。重要なのは方針そのものより、
**自動解決の判断を全て `AutoChoice` 1クラスに集約した**ことである。

```java
public static boolean shouldDrawOptional(PlayerState player) {
    return !player.getDeck().isEmpty();   // 原則YES、敗北するときだけNO
}

public static List<String> reviveOrder(CardMasterRepository cards, List<String> candidates) {
    // コストの高い順。同コストはID順で安定させる(実行のたびに結果が変わらないように)
}
```

各カードの効果は「誰が決めたか」を知らず、ただ `AutoChoice` に尋ねる。
将来 B案(割り込み選択)を実装するときは、**カード28枚には一切触れず**、
このクラスの呼び出し元だけをプレイヤーへの問い合わせに差し替えればよい。

技術的負債を作るなとは言わない。**作るなら、返済場所が1箇所になるように作る。**

## 4. カード実装で面白かったところ

### ゾンストライカーの無限ループ回避

```java
register("QTE-0012", TriggerType.ON_SUMMON, ctx -> {
    while (ctx.owner().getTrash().contains("QTE-0012") && !ctx.owner().isMinionZoneFull()) {
        ctx.owner().getTrash().remove("QTE-0012");
        ctx.actions().putIntoFieldByEffect(ctx.room(), ctx.owner(), "QTE-0012");
    }
});
```

`putIntoFieldByEffect` は**効果による「出す」であり、ON_SUMMON を発火しない**。
もしここが「召喚」だったら、出てきたゾンストライカーが自分の召喚時効果を発動し、
墓地が尽きるまで再帰する。**「召喚」と「出す」の区別が、そのまま無限ループの防波堤になっている。**

同名4枚制限の上書きは `DeckValidator.UNLIMITED_COPIES` に閉じ込めた。
カード側の性質だが、判定はデッキ検証でしか行えないためである。

### 冥界神ハデスの処理順序

```java
// ① ハデス以外の全ミニオンを破壊
// ② 裏向きマナの枚数を数える
// ③ 「このターン破壊された味方ミニオン」を墓地から出す
```

①で破壊したミニオンが、③の候補に**入る**。これは
Batch 10a で「破壊 → 墓地に入れる → 記録する」の順序を確定させたおかげで自然に成立した。
順序が逆なら、ハデスは自分が壊したものを蘇生できない。

なお、①の走査はリストのコピーに対して行うため、
不滅のネクロマンサーが①の最中に蘇生させたミニオンは破壊されずに残る。
「掃討が始まった後に登場したものは巻き込まれない」という解釈である。

### 悪夢のオーラ表現

「このターン、ミニオンの召喚コストを-4」は、既存の `thisTurnAuras` に
カードIDを積むだけで実現できた。ターン終了時のクリアも既存処理が面倒を見る。

```java
if (card.type() == CardType.MINION && owner.getThisTurnAuras().contains("QTE-0070")) {
    cost -= 4;
}
```

火文明の「溢れ出る英知」(手札枚数分の攻撃力バフ)のために作った仕組みが、
文明をまたいで再利用できた例である。

### 死者蘇生のコストが選択に依存する問題

「生贄にした数だけコスト-1」は、**支払い額が選択結果で決まる**初のカードである。
既存の流れは「検証 → コスト計算 → 支払い」なので、検証結果をコスト計算に渡す必要がある。

```java
if (SACRIFICE_SPELL_ID.equals(master.id())) {
    player.setPendingSacrificeCount(validated.minions().get(0).size());
}
try {
    payCost(player, stats.effectiveCost(state, player, master));
} finally {
    player.setPendingSacrificeCount(0);   // MP不足で弾かれた場合も必ず戻す
}
```

`finally` で戻すのが要点である。ここを忘れると、MP不足で失敗した次の使用時に
割引が残り続ける。**「一時的な状態」は必ず解除まで含めて1組で書く。**

## 5. 闇スターターデッキ

40枚(ミニオン28/ウェポン3/スペル9)。設計意図は「**自分のカードを壊して回す**」。

- 序盤: カース・ボーン(裏向きマナを作る)・ゾンストライカー5枚・生贄を求める邪鬼
- 中盤: 執念の暗殺者・不滅のネクロマンサー・ボーン・コレクターで、破壊をドローと蘇生に変換
- 終盤: 群がる死霊王(墓地のゾンストライカーで軽減)・冥界神ハデス

ゾンストライカーを**あえて5枚**入れてある。4枚制限の上書きが機能していることを
対戦の中で確認できるようにするためである。

## ✅ 動作確認手順

1. ロビーでリーダーに **冥府の禁皇** または **黄泉の召喚主** を選べることを確認
2. **カース・ボーン**を召喚 → マナが1枚裏向きになる(マナ表示で確認)。
   表向きマナが無い状態で召喚すると自壊する
3. **絶望の連鎖**(1コスト)→ 自分1体・相手1体を選んで両方破壊。
   自分のミニオンがいないときは使用できないことも確認
4. **執念の暗殺者**を出した状態で自分のミニオンを破壊 → 自動で1枚引く
5. **不滅のネクロマンサー**を出した状態で自分のミニオンを破壊
   → 裏向きマナが1枚減り、そのミニオンが【突進】付きで戻る
6. **墓穴の呪い** → 山札が3枚墓地へ行き、HPが墓地枚数以下のミニオンが全滅する
7. **禁忌の墓地利用** → 「墓地を開く」で墓地のスペルを2枚まで選び、マナが裏向きで増える
8. **死者蘇生** → 生贄を選ぶほどコスト表示が下がる。0体でも使える
9. **黄泉の召喚主**でサブフェイズに進む → 「墓地から召喚」ボタンが出る。
   ミニオンだけが一覧に並び、コストを払って召喚できる(【召喚時】も発動する)
10. **死神の大鎌**(Attack0)を装備してミニオンを攻撃 → ダメージ0でも破壊される
11. 水・火のデッキが今までどおり動くこと(回帰確認)

ブラウザキャッシュが残っていると新しいボタンが出ない。
`battle.js` は `v=10` に上げてあるので、通常は自動で読み直される。

## ✅ 理解確認

**問1.** ゾンストライカーの【召喚時】が `putIntoFieldByEffect`(効果で出す)を呼んでいる。
ここを `summonToField`(召喚)に変えると何が起きるか。

<details><summary>答え</summary>

出てきたゾンストライカーが自分の【召喚時】を発動し、また墓地から出す…を繰り返し、
墓地が尽きるかミニオンゾーンが埋まるまで再帰する。
ゾーン上限6体があるので厳密には無限ループにならないが、
1回の召喚で盤面が勝手に埋まる想定外の挙動になる。
</details>

**問2.** 「代償を払えないカードは使用できない」を、効果の中で例外を投げて
実現してはいけないのはなぜか。

<details><summary>答え</summary>

効果が実行される時点では、既にコストを支払い、カードを手札から取り除いている。
そこで例外を投げると、MPとカードだけを失って何も起きない状態になる。
拒否は必ず、状態を変更する前(`requirePlayable`)で行う必要がある。
</details>

**問3.** `AutoChoice` を独立したクラスにした利点を、
「後で割り込み選択システムを作る」という前提で説明せよ。

<details><summary>答え</summary>

自動解決の判断がカード28枚の効果に散らばっていると、
差し替えのために全カードを読み直して修正する必要がある。
1クラスに集約してあれば、呼び出し元(数箇所)をプレイヤーへの問い合わせに
置き換えるだけで済み、カードの効果には手を触れずに移行できる。
</details>
