# Batch 8 設計解説 — 火文明完成: 自傷というテーマを構造で支える

## ⚡ 結論チートシート

- **火文明28枚がすべて実カード通りに動く**。ロビーで火リーダーも選べるようになり、水↔火のどちらの構成でも対戦できる
- 火のテーマ(自傷)を支えるのは4つの土台: **被ダメージ回数カウンタ**、**ON_LEADER_DAMAGEDトリガー**、**発生源つきダメージ**、**回復回数カウンタ**
- 反転の炎鏡の「このカード以外の効果で」は、`damageLeader(…, sourceCardId)`の1引数で無限ループを止めている
- 【特殊召喚】が拡張: **MPコストを持てる**(ヴォルカニクス=1)、**特殊召喚限定の追加効果**を持てる(背水の炎壁)
- 対象フィルタを単一値から**条件リスト(AND)**へ。フレイム・スナイプ=【守護】かつHP5以下
- キーワード付与に**このターン限り**が加わった(捨て身の猛進の【突進】)

---

## 1. 「量」ではなく「回数」を数える ★

火文明の特殊召喚条件は、極炎竜ヴォルカニクス「リーダーが合計4回以上受けている」、背水の炎壁「3回以上ダメージを受けていた」、鳳凰神「5回以上回復した」。**すべて回数であって量ではない**。1回で10ダメージ受けても1カウント、1ダメージを4回受ければ4カウント。だからこそ火文明は「小刻みに自分を刻む」デッキになる——ルールの数え方がデッキの動きを決めている好例で、実装でも量と混同してはいけない。

カウンタは`PlayerState`に置き、`GameActions.damageLeader`と`healLeader`で加算する。ターン開始時の`startTurnReset`でリセット。**加算箇所が各1行で済んだのは、ダメージと回復の経路をずっと前から1本に集約してきたおかげ**である。もし「LPを直接引く」コードが各カードに散らばっていたら、20箇所以上にカウンタを足して回り、必ずどこかを忘れていた。

なお現時点で戦闘によるリーダーへのダメージ(`attack`内)は`setLp`を直接呼んでおり、カウンタとトリガーを通らない。カード効果由来のダメージのみが火のテーマに関与する、という切り分けである(炎鏡のテキストも「効果で」と限定している)。

## 2. 発生源つきダメージ — 無限ループをテキストが自分で止めている ★

反転の炎鏡: 「自分のターン中**このカード以外の効果で**自分のリーダーにダメージを与えられたとき、自分のリーダーに1ダメージ。その後1回復」。

この「このカード以外の」がなければ、炎鏡の与える1ダメージが炎鏡自身を再誘発し、無限ループでLPが尽きる。カードデザイナーが**テキストの中に停止条件を書き込んでいる**わけで、実装はそれを写すだけでよい——`damageLeader(room, player, amount, sourceCardId)`の第4引数である。

```java
boolean byOtherCard = sourceCardId != null && !"QTE-0048".equals(sourceCardId);
```

炎鏡が自分で与えるダメージには`"QTE-0048"`を渡すので、2周目で必ず止まる。一方、火炎の狂信者(被ダメージのたび攻撃力+2)には除外がないため、**炎鏡の1ダメージでも狂信者は成長する**。この非対称がそのままカードの強さの違いになっている。

トリガーの入口を`fire(TriggerType, MinionInstance, ctx)`とは別に`fireLeaderDamaged(ctx, sourceCardId)`として立てたのは、**発動源が場のミニオンとは限らない**からである(炎鏡は装備ウェポン)。「トリガーの主体はミニオン」という暗黙の前提が崩れた最初のケースで、ゾーン横断トリガー(水龍)に続く2例目の一般化になる。

## 3. 特殊召喚の一般化 ★

Batch 3で作った`SpecialSummonSpec`は「条件 + 代替コスト + 0コスト着地」の形だった。火文明が2つの穴を開けた。

- **ヴォルカニクス**: 「コストを**1にして**場に出せる」——0コストではない。`mpCost`フィールドを追加
- **背水の炎壁**: 「これで出したとき1回復。【召喚時】1回復」——特殊召喚で出したときだけ発生する効果と、通常の召喚時効果が**別々に書かれている**。`onSpecialSummon`を追加(特殊召喚なら計2回復、通常召喚なら1回復)

既存4枚の登録が壊れないよう、標準形(`mpCost=0`・追加効果なし)を作る`SpecialSummonSpec.of(...)`を用意して移行した。**新しい要求で仕様を広げるとき、既存の呼び出しを書き換えずに済むデフォルトを用意する**——ライブラリ設計でよく使う手で、水文明の登録は1文字も変えていない。

## 4. フィルタを「値」から「条件の集合」へ

フレイム・スナイプ「相手の【守護】を持つHP5以下のミニオン」で、対象の絞り込みが初めて複合条件になった。`Filter filter`(単数・nullable)を`List<Filter> filters`(AND条件)に変え、条件も5種に増やした(KNOWLEDGE / GUARD / MINION_CARD / HP_5_OR_LESS / COST_4_OR_LESS)。

判定で気をつけた点がひとつ。HP5以下は**現在HPで判定する**——ダメージを受けて弱った大型ミニオンも撃てるべきだからで、`checkFilter`は場のミニオンなら`MinionInstance`を、手札のカードなら`CardMaster`を見るように分けてある。同じ「HP」でも印刷値と現在値は別物、という区別はこのゲームでは何度も出てくる。

サーバとクライアントの両方に判定が必要になるが、**列挙の名前をそのままJSONに載せて共有**することで、条件の定義がRegistryの1箇所に留まっている(クライアントの`matchesFilters`は名前で分岐するだけ)。

## 5. 今回犯したミスの記録 ★

正直に書いておく。火文明のセクションを既存ファイルに差し込む際、**置換範囲を広く取りすぎて水文明の登録メソッド5個(スペル・トリガー・対象指定・特殊召喚・リーダー能力)を丸ごと消してしまった**。原因は、Batch 7で火のセクションをファイルの上部(コンストラクタ直後)に置いたため、「火の開始位置から次の見出しまで」という範囲指定が水の全登録を巻き込んだこと。

検知できたのは、パッケージングの前に**「デッキに入っている全カードの効果が登録されているか」を機械的に照合するチェック**を回していたから。「アクア・サーチが未実装」という、あり得ない結果が出て気づいた。人間の目視レビューでは、消えたコードは画面に出てこないので気づきにくい——**存在すべきものの一覧と突き合わせる検査**が効く場面である。

教訓は2つ。第一に、テキスト置換でコードを編集するときは**開始位置だけでなく終了位置も明示的に検証する**。第二に、ファイル内の配置は意味のまとまり順に保つ(火のセクションは今回、他の登録の後ろに移動した)。

## ✅ 動作確認手順

1. 差し替えて起動(静的ファイルにv=8を付けたのでキャッシュ問題は自動で解消される)
2. **ロビー**: リーダー選択に「水 / 蒼海の賢者」「火 / 傷痕の闘帝」など**4人**が並ぶ。片方を火リーダーにして開始
3. **自傷の連鎖**: 血誓のバーサーカー(自傷1、LP10以上なら追加2)→ 火炎の狂信者を出しておくと、自傷のたびに攻撃力が+2ずつ伸びる
4. **炎鏡**: 反転の炎鏡を装備して、イグニッション・バーストなどで自傷 → 「【反転の炎鏡】が反応した」ログの後、追加1ダメージ+1回復。**無限ループしない**ことを確認
5. **特殊召喚(MPあり)**: 自傷を4回してから極炎竜ヴォルカニクスをクリック → ★マークが出て、コスト**1**で出せる
6. **特殊召喚(追加効果)**: 自傷3回で背水の炎壁 → 特殊召喚だと1+1で2回復、通常召喚なら1回復
7. **複合フィルタ**: 相手の場に【守護】ミニオンがいる状態でフレイム・スナイプ → HP5以下の守護持ちだけが選択できる
8. **期限付き付与**: 捨て身の猛進 → 全ミニオンが攻撃+1と【突進】を得て、ターン終了で両方消える
9. **リーダー能力**: 剛火の将でライフ2減 → 次の火ミニオンのコストが1下がる(0にはならない)
10. **禁忌**: 火リーダーの禁忌デッキは**水文明8枚**に切り替わっている
