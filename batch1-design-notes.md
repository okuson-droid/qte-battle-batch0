# Batch 1 設計解説 — STOMPの構造とMVCとの対比

## ⚡ 結論チートシート

- WebSocketは「接続しっぱなしの土管」、STOMPはその上の「宛先付きメッセージの規約」
- MVCとの対応: `@GetMapping("/url")` ↔ `@MessageMapping("/room/{id}/summon")`。ただし**戻り値で返さない**。サーバは好きなタイミングで好きな宛先に送る
- 盤面は`GameState`をそのまま送らず、**プレイヤーごとにフィルタしたDTO**(`GameViewBuilder`)を`/topic/room/{id}/player/{playerId}`へ送り分ける
- 同時操作は`synchronized (room.getLock())`で「1部屋1操作」に直列化(`GameWsController.execute`)
- クライアントは状態を持たない**再描画方式**: 届いた最新ビューが常に正
- 動作確認: 2タブで部屋を作成/入室 → ダイス → 先後選択 → マナチャージ → 召喚 → 攻撃 → LPを削り切ると決着

---

## 1. MVCの往復モデルと、その限界 ★

EM-Springで学んだMVCは「クライアントが尋ね、サーバが答える」の往復である。重要なのは**会話を始められるのが常にクライアント側だけ**という点だ。サーバは聞かれるまで黙っているしかない。

対戦ゲームではこれが致命傷になる。相手がカードを出した瞬間、**あなたは何もしていない**。何もしていないあなたの画面を書き換えるには、サーバから話しかける手段が要る。それがWebSocketである。最初にHTTPで「これからWebSocketに切り替えます」という握手(ハンドシェイク、本アプリでは`/ws`)をしたあと、接続は張りっぱなしになり、以降は**どちらからでも、何度でも**メッセージを送れる。

## 2. STOMP = WebSocketの上の「宛先」の規約 ★

素のWebSocketは土管であり、流れるデータに意味づけがない。STOMPはそこに「宛先(destination)」という概念を足した薄い規約で、Springのサポートもこの単位で書ける。本アプリの宛先は3種類しかない。

| 宛先 | 方向 | MVCでの対応物 |
|---|---|---|
| `/ws` | 接続時のみ | (なし: 電話をかける行為) |
| `/app/room/{id}/summon` など | クライアント→サーバ | リクエストURL。`@MessageMapping`が受ける |
| `/topic/room/{id}/player/{playerId}` | サーバ→クライアント | **対応物なし**。購読者への配信 |

`@MessageMapping`のメソッドが`@GetMapping`と決定的に違うのは**戻り値がない**ことだ。MVCでは「リクエストの戻り」が唯一の返答手段だったが、STOMPでは処理の結果を`SimpMessagingTemplate`(`GameBroadcaster`)で**任意の宛先に、任意の数だけ**送る。実際、召喚1回の処理はメッセージ2通(両プレイヤーへのビュー配信)を生む。1入力→N出力。ここがMVCの往復と一番違う頭の切り替えどころである。

クライアント側(battle.js)は逆向きの同じ構造で、`subscribe`(受信箱の購読)と`publish`(宛先への送信)だけでできている。

## 3. プレイヤーごとに違う世界を見せる — ビューのフィルタリング ★

カードゲームには非公開情報がある。相手の手札、相手の裏向きマナ。もし`GameState`をそのままJSONで配れば、ブラウザの開発者ツールを開くだけで相手の手札が丸見えになる。**画面に表示しないことと、データを送らないことは別問題**であり、守れるのは後者だけである。

そこで配信の直前に`GameViewBuilder`が視点ごとのDTOを組み立てる。

- 自分のビュー: 手札は中身(`hand`)、裏向きマナも中身入り
- 相手のビュー: 手札は`handCount`(枚数)のみで`hand`は`null`、裏向きマナは`cardId=null`

同じ試合から**2つの異なるJSON**が生まれ、`/topic/room/{id}/player/{playerId}`へ別々に送られる。宛先に含まれるplayerIdは推測不能なUUIDなので、これが実質的な「本人だけの受信箱」になる(本格的な認証の代替としての、趣味アプリの割り切り。仕様1-2)。

EM-Springでも「EntityをそのままViewに渡さずFormやDTOに詰め替える」を学んだが、あちらの動機は主に画面都合とバリデーションだった。今回の動機は**セキュリティ(情報隠蔽)**である。同じ「詰め替え」でも理由が違う、というのは設計の引き出しとして覚えておく価値がある。

## 4. 同時操作から状態を守る ★

Batch 0の解説で予告した`room.getLock()`が、今回から実際に働く。両プレイヤーの操作は別スレッドでほぼ同時に届きうる。たとえば「Aの攻撃」と「Bのターン終了(不正)」が同時に走ると、検証と変更の間に状態が変わる競合が起きる。

`GameWsController.execute()`がすべての操作を共通の型に流し込む:

```
synchronized (room.getLock()) {   // 1部屋につき同時1操作
    action.apply(room);           // GameServiceがルール検証と状態変更
}
broadcaster.broadcast(room);      // 成功したら両者へ配信
```

例外(ルール違反)なら状態は変更されておらず、**操作した本人にだけ**エラーを返す。相手の画面には何も起きない — 不正操作の試みすら相手には見せない、という情報設計でもある。

もうひとつ大事な原則: **正当性の最終判定は必ずサーバで行う**。クライアントの「攻撃できそうなカードの光り方」(`canAttackMinion`等)はただのUI補助で、改造クライアントがどんなメッセージを送ってきても`GameService`の検証(`validateAttack`ほか)が弾く。JSの検証は突破できるがサーバの検証は突破できない — EM-Springのバリデーション(クライアント側とサーバ側の二重化)と同じ構図である。

## 5. クライアントは「描き写す」だけ — 再描画方式

battle.jsは差分更新をしない。ビューが届くたび盤面全体を描き直す。「攻撃で相手のHPだけ書き換える」ような差分方式は速いが、更新漏れのバグ(サーバと画面のズレ)を生みやすい。この規模なら全再描画のコストは無視でき、**「サーバから届いた最新ビューが常に正」という単純さ**を優先した。クライアントが自分で持つ状態は「攻撃元として選択中のミニオン」ただ1つである。

## 6. Batch 1に意図的に「まだ無い」もの

- **スペル・ウェポンのプレイ**: エフェクトシステム(Batch 2)待ち。プレイしようとすると拒否メッセージが出る。ピュア・エレメントも同様に手札に来るが使えない
- **【召喚時】【攻撃時】等のトリガー効果・リーダー起動能力・還元**: Batch 2
- **【知識】だけ先行実装**: 登場時1ドローの1行で済み、水デッキの体感に不可欠なため。GameService内にTODOを付けてあり、Batch 2でエフェクトシステムへ移植する(ad hoc実装→仕組みへの吸収、というリファクタリングを見せる意図もある)
- **マリガン**: セットアップの複雑さを抑えるため後回し
- **山札切れ=敗北は仮ルール**: 総合ルールに規定がないため慣例を仮採用した。→ **要確認**

## ✅ 動作確認手順

1. 差し替え: プロジェクトを丸ごと今回のzipの内容で置き換える(Batch 0からの変更ファイルが多いため)
2. 起動して http://localhost:8080 → ロビーが表示される
3. タブ1: 名前とリーダーを選んで「部屋を作成して入室」→ 画面上部の**部屋コード**を控える
4. タブ2(シークレットウィンドウでなくてよい): ロビーから部屋コードで「入室する」
5. ダイス結果がログに出て、勝った側のタブに先攻/後攻ボタンが出る
6. 手札クリックでマナチャージ(マナチャージフェイズ)→「次のフェイズへ」→ 手札クリックで召喚(メインフェイズ)→ バトルフェイズで自分のミニオン→相手ミニオンかリーダーの順にクリックで攻撃
7. 【知識】持ちの召喚でドローされること、【守護】がいると他を攻撃できないこと、召喚したターンのミニオンが攻撃できないこと(突進はミニオンにだけ可能なこと)を確認
8. LPを削り切ると決着メッセージ
