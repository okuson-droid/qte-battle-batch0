# Batch 3 設計解説 — 対象指定: 「選ぶ」をクライアントとサーバで分担する

## ⚡ 結論チートシート

- 「何を選ばせるか」の定義(`TargetSpec`)はRegistryに1箇所だけ置き、クライアントは選択UIの進行に、サーバは検証に、**同じ定義**を使う
- クライアントは選択を集めて送るだけ。正当性(潜伏・知識フィルタ・重複・枚数)の最終判定は`validateTargets`が行う——UIの絞り込みは操作補助にすぎない
- 処理順序の鉄則: **検証(状態を変えない)→ 支払い → 手札除去 → 着地 → 効果**。検証で弾かれたとき状態が無傷であることを保証する
- 手札を複数枚取り除くときは**大きいインデックスから**消す(小さい順だと後続の位置がずれる)
- 【特殊召喚】は「条件＋代替コスト＋0コスト着地」の別ルートで、着地後は通常召喚と同じ`summonToField`に合流する(ON_SUMMON/ON_ENTER両方発動=召喚扱いの裁定通り)
- 動作確認: 大蟹の2段階選択、継承者の「選ばない」、双流のコスト割引表示、特殊召喚4種

---

## 1. 対象指定はプロトコル設計の問題 ★

「相手のミニオン1体に○○する」を実装するには、**プレイヤーの選択**という入力が要る。Batch 2までの操作(召喚・攻撃)は1クリック=1メッセージで済んだが、大蟹は「捨てる手札」と「戻す相手ミニオン」の**2段階の選択**を要求する。これをどう運ぶかは設計の分かれ道で、大きく2案ある。

- 案A: サーバ主導。効果の処理を途中で止め、「選んでください」とクライアントに問い合わせ、答えが来たら再開する
- 案B: クライアント先集め。プレイ前に必要な選択を全部集め、**1つのメッセージに同梱**して送る

案Aはルール表現として自然だが、「実行途中の効果」という中断状態をサーバが保持する必要があり、切断・不正・タイムアウトの処理が一気に難しくなる。本アプリは案Bを採った。サーバは常に「メッセージを受けて、検証して、最後まで実行する」だけ——**処理の途中状態が存在しない**ので、Batch 1以来の「1部屋1操作のsynchronized」モデルがそのまま保てる。

案Bの前提は「クライアントが、何を選ばせるべきか知っている」こと。そこで`TargetSpec`(要求の列: 種類・陣営・数・任意か・絞り込み・案内文)をRegistryに定義し、`CardView`に載せてクライアントに配る。battle.jsの選択フロー(`pending`)はこの仕様を読んで進行するだけで、**カード固有の選択ロジックをJSに1行も書いていない**。カードを追加してもJSは変わらない——Batch 2でGameServiceに敷いた「カードを知らない」原則を、クライアントにも延長した形である。

## 2. それでも最終判定はサーバ ★

クライアントは潜伏や知識フィルタで選択肢を絞るが、それは操作補助であって防壁ではない。改造クライアントは任意のinstanceIdを送れる。`validateTargets`が全要求を検証する: 枚数(任意なら0かN)、範囲、重複、プレイするカード自身の除外、知識フィルタ、そして**【潜伏】=相手の効果の対象にならない(自分は対象にできる)**。この非対称な検証は「owner != player かつ STEALTH」という1行の条件に落ちている——仕様確定フェーズの裁定がコードの条件式に直訳される好例なので見てほしい。

もうひとつの鉄則が処理順序である。`validateTargets`は**状態を一切変更しない**。支払い・手札除去・効果はすべて検証通過後。これにより「拒否された操作は何も起こしていない」が常に成り立つ(GameWsControllerがエラーを操作者だけに返す設計は、この保証の上に載っている)。

## 3. インデックスは揮発する — 手札除去の落とし穴 ★

地味だが実戦的な論点。クライアントが送る「手札の3枚目」は、**送った瞬間の手札**に対する位置である。サーバが先にプレイするカード本体を手札から抜くと、それより後ろのカードの位置が1つずつ前にずれ、選択が別のカードを指してしまう。

対策は2つの手順に現れている。①除去前に全対象の**カードIDを確定**させる(`removePlayedAndTargets`冒頭)。位置は揮発するがIDは揮発しない。②除去は**大きいインデックスから**行う。小→大の順で消すと1回消すたびに残りの位置がずれるが、大→小なら消した位置より前は影響を受けない。EM-Springで学んだ「Listをループ中に削除してはいけない」問題の親戚であり、この種のオフバイワンは動作確認で表面化しにくい(選んだカードと違うカードが捨てられても気づきにくい)ので、設計段階で潰すべき類のバグである。

## 4. 特殊召喚 — 別ルートで入り、同じ場所に着地する ★

【特殊召喚】の定義(確定済み)は「カード記載の条件・代替コストで手札から出せる代替召喚。召喚として扱う」。実装はこの文の構造をなぞっている。

- **条件・代替コスト・説明文**は`SpecialSummonSpec`としてRegistryに登録(カードごとに違う部分)
- 入口`specialSummon`は通常の`playMinion`と別メソッド(MPを払わない・条件を検証する・代替コストを支払う)
- 着地は両者とも`summonToField`に合流し、ON_SUMMON/ON_ENTERの両方が発火する——「召喚として扱う」の裁定がメソッド共有として表現される

4枚の仕様がそれぞれ設計の課題を1つずつ持ち込んだ: プレサージュは「手札の知識5枚」の条件で**カード自身を数えから除外**する必要(条件関数がhandIndexを受け取る理由)。双翼は代替コストが自分のミニオン2体を場から離すため、**ミニオンゾーン上限の事前チェックを免除**する(バウンスで枠が空くから)。ポセイドン・コアは付与キーワード(`grantedKeywords`)の初の実用。海皇の「メインフェーズ開始時」は厳密には時点指定だが、「このターンまだカードをプレイしていない」(`playedCardThisTurn`)で**近似**した——同値ではない(マナチャージは数えない等)ことを明記しておく。厳密化が必要になったら要相談。

## 5. 動的コストがUIに滲み出る

双流の幻術師(場の知識の数だけCost-1、両者の場を参照)で、Batch 2の「保存せず評価する」方針がコストにも適用された(`StatCalculator.effectiveCost`)。面白いのはUI側で、手札の表示が`(7)→5`のように割引後を見せる。実効コストはサーバが計算して`CardView.effectiveCost`で配る——クライアントに計算式を持たせると、サーバと式がズレたとき「表示上は出せるのに拒否される」という混乱を生むからである。**導出値の計算場所は1箇所**、はここでも守られている。

## 6. 意図的に「まだ無い」もの(Batch 4 = 水文明完成)

ウェポン装備とリーダー攻撃(三叉槍・氷結の杖の凍結・水刺客の動的攻撃力)、リーダー起動能力(賢者・智者)、ピュア・エレメント(一時マナ)、黄泉還る水龍の墓地トリガー(マナが離れるイベントはBatch 4のリーダー能力とピュア・エレメントで初めて発生するため、一緒に実装するのが自然)、マリガン。

## ✅ 動作確認手順

1. 差し替え(src消去→展開→F5→Maven Update→Clean)して起動、2タブで対戦開始
2. **2段階選択**: 手札を喰らう大蟹をプレイ → 「捨てるカードを選んでください」→手札クリック → 「手札に戻す相手のミニオンを…」→相手ミニオンクリック → 両方が解決される
3. **任意選択**: 英知の継承者 → 「選ばない」ボタンで捨てずに召喚できる。捨てた場合は1ドローされる
4. **潜伏の保護**: 相手の黄泉還る水龍(潜伏)は大蟹の対象に選べない(自分の水龍は双翼の対象に選べる)
5. **動的コスト**: 場に知識ミニオンが並ぶと、手札の双流の幻術師のコスト表示が下がる
6. **特殊召喚**: 手札に★マークが出たら特殊召喚可能。クリックで確認ダイアログ → 双翼(自分の知識ミニオン2体を戻す)、プレサージュ(知識5枚を山札の下へ)、コア(条件のみ・全員が突進を得て即攻撃できる)、海皇(ターン最初のプレイでのみ可・3枚捨てる)
7. **キャンセル**: 選択の途中でキャンセルすると何も起きていない(コストも払われていない)

## ✅ 理解確認(2問)

1. `validateTargets`を通過した後、`payCost`の前に相手の操作が割り込んで対象のミニオンが場を離れる——という事故が起きない理由は何か(Batch 1の解説の内容と繋げて答えよ)。
2. 手札`[A, B, C, D]`でBをプレイし、CとDを捨てる選択(インデックス2, 3)を送ったとする。除去を小さいインデックス順(1→2→3)で行うと、実際に手札から消えるのはどのカードか。
